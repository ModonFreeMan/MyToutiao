# 今日头条视频播放流 + 搜索模块 MVP 技术方案

> 声明：本文档中的架构设计、模块设计、流程设计以及对应项目实现均为 AI 辅助生成。

## 1. 技术栈


| 类别   | 选型                                  |
| ---- | ----------------------------------- |
| 开发框架 | Flutter                             |
| 状态管理 | Riverpod                            |
| 播放能力 | Flutter 视频播放器插件 / 系统播放器封装           |
| 本地存储 | SharedPreferences                   |
| 数据来源 | 本地 Mock 数据 / 本地缓存 / 搜索离线索引 / 向量检索服务 |


## 2. 总体架构设计

项目采用 Flutter + Riverpod 的分层架构。

```text
Page / Widget 页面层
        ↓
Coordinator / Orchestrator 业务编排层
        ↓
ViewModel / Notifier 状态管理层
        ↓
UseCase / Service 业务服务层
        ↓
Repository 数据仓库层
        ↓
DataSource 数据源层
```

### 2.1 分层职责

#### 页面层

负责 UI 展示、用户交互、页面跳转。

#### 业务编排层

负责跨模块业务流程编排，例如内容流当前项变化后的播放 / 停止决策、搜索结果点击后的 Feed 定位与播放器触发、Feed 被搜索页覆盖前后的暂停 / 恢复。

#### 状态管理层

负责页面状态维护、事件处理、异步数据加载。

#### 业务服务层

负责播放器控制、推荐词匹配、搜索匹配、性能统计等业务逻辑。

#### 数据仓库层

负责统一封装数据访问，对上层屏蔽数据来源。

#### 数据源层

负责提供本地 Mock 数据、本地缓存数据、搜索离线业务文档和向量检索结果。

#### 推荐词匹配优先级

4. 如果 title 没有命中，则返回默认推荐词

### 4.10 observability 模块

#### 职责

- 旁路记录业务链路中的可观测事件
- 维护起播、预加载、首帧、缓冲等 session 归属
- 处理异步回调迟到、重复事件、失败事件的归属校验
- 输出稳定的调试报告和 baseline report
- 为 performance 模块提供可统计的原始观测数据
- 不参与播放调度决策
- 不直接控制播放器生命周期
- 采集失败不能影响正常业务流程
- Debug report 仅作为调试和测试观测入口，不参与业务决策

#### 核心服务

```text
PlaybackStartupMetrics
PlaybackStartupSession
PlaybackStartupReport
PlaybackStartupDebugReport
```

#### 核心观测事件

```text
feed_item_visible：Feed 当前项进入可见状态，作为本次起播 session 起点。
controller_initialize_start：播放器 controller 开始初始化。
controller_initialize_end：播放器 controller 初始化成功结束。
controller_initialize_failed：播放器 controller 初始化失败。
first_frame_rendered：播放器画面首次进入渲染树，作为 Flutter 层近似首帧。
play_requested：业务发起播放请求。
actual_playing：播放器确认进入播放状态。
buffering_start：播放器开始缓冲。
buffering_end：播放器结束缓冲。
preload_start：后台 preload controller 开始初始化候选视频。
preload_ready：候选视频 preload 初始化完成，可等待命中。
preload_failed：候选视频 preload 初始化失败。
preload_hit：当前可见视频命中已准备好的 preload controller。
preload_miss：当前可见视频需要 preload 但未命中可提升 controller。
preload_promoted_to_active：preload controller 被提升为 active controller。
preload_disposed_before_hit：preload controller 在命中前被释放或覆盖。
```

#### 设计原则

```text
Observability 只能观察业务流程。
Observability 不能决定 active / preload / dispose。
Observability 不能修改用户播放意图。
Observability 不能触发 UI 高频 rebuild。
所有观测事件必须绑定 session / token 并在内部完成归属校验。
迟到事件、重复事件、失败事件必须被安全记录或忽略，不能抛出影响播放链路。
```

## 6. Riverpod 状态管理设计

Riverpod 负责依赖注册、状态托管、异步结果派生和跨模块编排入口。

### 6.1 Provider 职责

#### 依赖注册

```text
sharedPreferencesProvider：提供 SharedPreferences 实例。
storageServiceProvider：封装轻量本地存储读写。
searchHistoryServiceProvider：管理搜索历史关键词。
feedDataSourceProvider：提供内容流数据源。
feedRepositoryProvider：封装内容流数据访问。
searchDataSourceProvider：提供搜索数据源。
searchRepositoryProvider：封装搜索数据访问。
recommendationDataSourceProvider：提供推荐词数据源。
recommendationRepositoryProvider：封装推荐词数据访问。
```

负责注册本地存储、DataSource 和 Repository，对上层屏蔽具体数据来源。

#### 状态托管

```text
feedViewModelProvider：维护内容流列表、分页和当前索引状态。
playerControllerProvider：维护播放器状态和播放器实例生命周期。
searchViewModelProvider：维护搜索输入、搜索历史和提交状态。
```

负责维护 Feed、Player、Search 三类可变状态。

#### 编排与服务

```text
feedPlaybackCoordinatorProvider：编排 Feed 当前项变化后的播放、停止和预加载。
feedPreloadEnabledProvider：控制 Feed 预加载能力开关。
recommendationServiceProvider：根据当前内容匹配推荐词。
playbackStartupMetricsProvider：采集播放起播和预加载观测事件。
playbackStartupDebugReportProvider：导出调试用播放指标报告。
performanceServiceProvider：统计起播性能和优化效果。
```

负责播放编排、预加载开关、推荐词匹配、观测指标和性能统计。

#### 异步派生

```text
searchResultsProvider：根据搜索词异步加载视频搜索结果。
recommendationWordsProvider：根据当前 FeedItem 异步加载推荐词。
```

负责根据搜索词或当前内容派生异步结果，不写入页面主状态。

### 6.2 关键状态流程

#### Feed 流程

```text
FeedPage：内容流页面
  ↓ watch
feedViewModelProvider：内容流状态入口
  ↓
FeedState：内容流页面状态
  ↓
UI rebuild：刷新内容流 UI
```

FeedViewModel 管理内容列表、分页、当前索引和搜索结果定位状态。FeedPage 根据 FeedState 重建 UI，并在当前项变化时通知 FeedPlaybackCoordinator 编排播放、停止和预加载。

#### Player 流程

```text
VideoFeedCard / LandscapePlayerPage：播放器展示页面
  ↓ watch
playerControllerProvider：播放器状态入口
  ↓
PlayerState：播放器页面状态
  ↓
播放 UI 更新：刷新画面、进度和控制层
```

PlayerController 管理 active 播放器、preload 播放器、播放意图、进度、清晰度和横屏渲染状态。播放命令来自播放器 UI 或 FeedPlaybackCoordinator。

#### Search 流程

```text
SearchMiddlePage / SearchResultPage：搜索页面
  ↓ watch
searchViewModelProvider：搜索状态入口
  ↓
SearchState：搜索输入和历史状态
  ↓
搜索输入和历史 UI 更新：刷新输入框、历史列表和提交状态
```

```text
SearchResultPage：搜索结果页面
  ↓ watch
searchResultsProvider(keyword)：搜索结果异步入口
  ↓
SearchRepository：搜索仓库
  ↓
AsyncValue<List<VideoFeedItem>>：搜索结果加载状态
```

SearchViewModel 管理关键词、搜索历史和提交状态；搜索结果由 searchResultsProvider 按 keyword 派生。搜索结果点击后通过 FeedPlaybackCoordinator 回到 Feed 并定位播放。

#### Recommendation 流程

```text
RelatedSearchEntry：相关搜索入口
  ↓ watch
recommendationWordsProvider(item)：推荐词异步入口
  ↓
RecommendationService：推荐词匹配服务
  ↓
AsyncValue<List<String>>：推荐词加载状态
```

推荐词按当前 FeedItem 派生，不进入 FeedState。

#### Observability 流程

```text
FeedPlaybackCoordinator / PlayerController / VideoPlayerView：播放链路事件来源
  ↓ read
playbackStartupMetricsProvider：播放观测指标入口
  ↓
PlaybackStartupMetrics：起播和预加载事件收集器
  ↓
PlaybackStartupReport / DebugReport：统计报告和调试报告
```

Observability 只旁路记录播放链路事件并输出报告，不控制播放器生命周期，不修改用户播放意图。

```text

  ↓
```

## 11. 搜索结果匹配策略

### 11.1 搜索范围

```text
SearchVideoDocument.title
SearchVideoDocument.summary
SearchVideoDocument.keywords
VideoFeedItem.id
```

### 11.2 匹配规则

当前默认数据源为 DenseSearchDataSource。

```text
用户搜索词
  ↓
OpenAI-compatible embedding service 生成查询向量
  ↓
MilvusSearchVectorStore 召回向量结果
  ↓
JsonSearchBusinessStore 读取离线业务文档
  ↓
关键词分数与向量分数融合排序
  ↓
根据 videoId 映射回 VideoFeedItem
```

融合分数：

```text
finalScore = vectorScore * 0.7 + normalizedKeywordScore * 0.3
```

关键词分数：


| 命中位置           | 分数  |
| -------------- | --- |
| title 包含搜索词    | +5  |
| keywords 包含搜索词 | +4  |
| summary 包含搜索词  | +1  |


### 11.3 搜索结果过滤

- 只返回视频类型内容
- 图片卡不出现在搜索结果中
- 融合分数大于 0 的视频进入结果列表
- 结果按照分数从高到低排序
- 如果向量检索、配置或离线业务文档不可用，则 fallback 到 MockSearchDataSource

### 11.4 搜索离线索引生成

当前搜索模块除运行时检索外，还保留离线索引构建能力。

```text
OfflineVideoInput
  ↓
VideoSummaryGenerator
  ↓
SearchVideoDocument
  ↓
VideoSearchOfflineIndexer
  ↓
SearchBusinessStore / SearchVectorStore
```

离线索引构建负责把视频基础信息转换为可检索业务文档，并生成摘要、关键词和向量写入配置指定的本地业务文档或向量存储。运行时 `DenseSearchDataSource` 读取这些离线产物完成召回与排序；如果配置、业务文档或向量服务不可用，则回退到 Mock 搜索数据源。

## 12. 性能优化设计

### 12.1 起播速度优化目标

- 降低首个视频首帧耗时
- 降低上下滑切换视频后的起播耗时
- 降低视频初始化等待时间
- 提升预加载命中率
- 通过数据展示优化幅度

### 12.2 优化手段

- 当前视频优先初始化
- 下一个视频提前预加载
- 复用已预加载的播放器实例
- 快速滑动时取消过期预加载任务
- 图片卡不创建播放器
- 分页数据提前加载
- 封面在播放器初始化前作为占位展示

### 12.3 播放器缓存策略

```text
当前索引：currentIndex
active controller：当前正在播放的视频
preload controller：单槽位，仅保留一个候选视频
候选选择：根据滑动方向优先选择后一个或前一个视频
释放策略：Feed 数据替换、候选失效、开关关闭或新任务覆盖时释放 preload
```

### 12.4 职责边界

```text
FeedPlaybackCoordinator
- 在 currentIndex 变化后编排当前视频播放、方向感知预加载、过期 preload 取消和页面覆盖暂停 / 恢复

PlayerController
- 管理 active controller 和单槽位 preload controller
- 命中 preload 时 promote 为 active controller
- 幂等释放 active / preload 播放器实例

Observability
- 旁路记录 Feed 起播、播放器初始化、首帧、播放确认、缓冲、预加载命中 / 失败等事件
- 维护 session / token 归属，忽略迟到或重复事件
- 不参与 active / preload / dispose 决策
- 不直接控制播放器生命周期

PerformanceService
- 消费 Observability 输出的观测数据或 report
- 统计优化前后起播耗时和优化幅度
- 不直接控制播放器生命周期
```

### 12.5 性能统计指标

```text
videoInitStartTime
videoReadyTime
firstFrameRenderTime
switchStartTime
playStartTime
preloadStartTime
preloadReadyTime
preloadHit
preloadMiss
preloadPromotedToActive
```

### 12.6 优化数据展示

技术文档中展示以下数据：

- 未优化平均起播耗时
- 优化后平均起播耗时
- 预加载命中率
- 预加载提升为 active 的次数
- 平均节省耗时
- 优化幅度百分比

当前采样结果：

```text
采样时间：2026-06-02
数据来源：
  baseline: build/reports/baseline.json
  preload only: build/reports/preload.json

  p50: baseline 53ms -> preload only 0ms，下降 53ms
  p90: baseline 686ms -> preload only 0ms，下降 686ms
  p95: baseline 686ms -> preload only 0ms，下降 686ms

  p50: baseline 252ms -> preload only 204ms，下降 48ms
```

- Integration 测试覆盖基础播放、完整视频流、搜索结果回 Feed 定位、搜索与清晰度、横屏播放、预加载指标采集等端到端流程。
- `helpers` 提供测试 App、ProviderContainer 创建和 fake video player platform，避免测试依赖真实播放器环境。

## 16. 架构总结

本项目使用 Flutter + Riverpod 实现，整体采用分层架构。

核心模块包括：

- `feed`：负责内容流展示、分页、视频卡、图片卡
- `player`：负责播放、暂停、进度、清晰度、横屏、预加载和 preload promote
- `search`：负责搜索中间页、搜索历史、搜索结果和混合检索入口
- `feed/coordinators`：负责跨模块业务流程编排，例如 Feed 播放切换、搜索结果定位播放、页面覆盖时的暂停 / 恢复
- `recommendation`：负责相关搜索推荐词匹配
- `observability`：负责播放链路可观测事件采集、session 归属校验和 baseline report
- `performance`：负责起播性能统计和优化幅度计算
- `storage`：负责本地搜索历史和轻量配置存储
- `data`：负责数据模型、仓库和数据源
- `core`：负责全局基础能力
- `common`：负责通用组件

整体数据流保持单向依赖：

```text
UI Page
  ↓
Coordinator / Orchestrator
  ↓
Riverpod ViewModel / Notifier
  ↓
Service / Repository
  ↓
DataSource
```
