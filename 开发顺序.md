# 第 5 阶段：基础播放器能力

### 目标

视频卡可以播放、暂停、显示进度。

### 开发内容

```text
1. 接入 video_player 或其他播放器插件
2. 创建 PlayerController
3. 创建 PlayerState
4. 创建 FeedPlaybackCoordinator
5. FeedPage 当前项变化时通知 FeedPlaybackCoordinator
6. FeedPlaybackCoordinator 根据 FeedItem 类型决定播放 / 停止
7. 当前视频进入页面时由编排层触发播放器初始化
8. 离开当前页面时由编排层触发暂停 / 释放播放器
9. 点击视频区域暂停 / 播放
10. 显示进度条
11. 支持拖动进度条

```


### 完成标准


# 第 12 阶段：起播速度优化

这个阶段不要太早做。等基础播放链路稳定后再做，否则容易调不清楚问题。

### 目标

实现预加载、预渲染，并通过数据说明优化效果。

### 开发内容

```text
1. 当前视频播放时，提前初始化下一个视频
2. 下一个视频提前加载资源
3. 下一个视频准备首帧
4. 用户滑到下一个视频时直接复用
5. 只保留 currentIndex - 1、currentIndex、currentIndex + 1 的播放器
6. 释放远离当前 index 的播放器

```

职责边界：

```text
由 FeedPlaybackCoordinator 在 currentIndex 变化后触发当前视频播放、下一个视频预加载 / 预渲染、远离当前 index 的播放器释放。
PlayerController 负责播放器实例缓存、复用和释放。
PerformanceService 只负责记录耗时、命中情况和优化幅度，不直接控制播放器。

```

### 先写这些文件

```text
lib/features/performance/services/performance_service.dart
lib/features/performance/models/performance_record.dart
lib/features/performance/providers/performance_provider.dart
lib/features/player/controllers/player_controller.dart

```

### 完成标准

```text
可以记录优化前后的起播耗时。
可以给出类似下面的数据：

未优化平均切换起播耗时：850ms
优化后平均切换起播耗时：320ms
优化幅度：62.35%

```

# 实际编码时的优先级

```text
App 启动
  ↓
FeedPage
  ↓
上下滑内容流
  ↓

视频播放
  ↓

  ↓
```

这条链路必须最先跑通。

# 最小可运行版本应该先做到什么程度

你第一版不要追求完整，先做到：

```text
1. App 能启动
2. 进入 FeedPage
3. 有 5 条 Mock 数据
4. PageView 可以上下滑
5. 视频卡可以播放
6. 图片卡可以展示
7. 顶部搜索框可以跳转搜索页
8. 当前内容变化时由业务编排层触发播放 / 停止

```

这就是你的第一个里程碑。

之后再逐步叠加：

```text
搜索结果 → 推荐词 → 分页 → 清晰度 → 横屏 → 预加载 → 性能统计

```

这样开发风险最低，也最容易调试。
