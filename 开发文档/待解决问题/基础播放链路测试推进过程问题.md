# 基础播放链路测试推进过程问题

## 问题背景

本次目标原本是：

```text
先添加基础播放链路的整体测试，
用测试暴露当前播放编排问题，
再基于暴露出的失败点决定实现修改。
```

新增测试确实先暴露了一个明确问题：

```text
用户从首个视频页滑到图片页后，
页面已经展示图片内容，
FeedViewModel.currentIndex 已经变为 1，
但 PlayerController.videoId 仍然是 video_001。
```

失败断言为：

```text
Expected: null
Actual: 'video_001'
```

这个失败点说明：Feed 当前项切换到非视频内容后，播放器停止编排没有稳定落地。

更准确地说，根本问题可以总结为：

```text
播放器控制器的业务状态生命周期和底层播放器资源生命周期没有解耦，
导致 Feed 已经切到非视频项时，业务层仍暴露旧 videoId。
```

Feed 编排层需要的是“当前页面是否应该有 active video”的同步结果；而 `PlayerController.stop()` 当前却把这个业务结果延迟到 `dispose` 完成之后。`dispose` 是底层播放器资源释放细节，具有异步性和平台差异，不应该阻塞业务状态清空。

因此，最小正确修复路径不应该先改 `FeedPage`，也不应该先调整 fake video platform，而应该先用 `PlayerController` 单元测试证明 stop 的状态机语义问题，再把 stop 改成：

```text
先业务停止，后资源释放。
```

## 本次推进中的错误

在测试仍处于红灯阶段时，过早修改了实现代码和测试辅助代码。

这导致多个问题混在了一起：

- 原始业务问题：Feed 当前项切到图片页后播放器未停止。
- 测试平台问题：fake video platform 的 dispose / stream 行为可能导致测试卡住。
- 状态机问题：`PlayerController.stop()` 是先等待底层 dispose，再清空业务状态，测试读取状态时可能仍是旧视频。
- 测试断言问题：测试曾要求 fake 平台的 `disposeCount` 立即变为 1，但底层资源释放本身是异步细节，不一定适合作为基础播放链路的核心断言。

这些问题并不是全部都通过独立测试先暴露出来，而是在调试过程中被直接修改了。

这不符合本轮应遵守的节奏：

```text
先写测试
先稳定复现问题
先记录失败
再决定是否修改实现
```

### 3. FeedPage 播放监听改动

文件：

- [feed_page.dart](d:/Flutter/VideoPlayer/lib/features/feed/pages/feed_page.dart:21)

改动内容：

```text

改为：

  )
```

设计目标：

```text
PageView.onPageChanged 只负责 setCurrentIndex；
当前 Feed item 变化统一触发 FeedPlaybackCoordinator.handleFeedCurrentChanged。
```

这符合优化文档中的最终设计，但当前问题是：该实现修改是在测试红灯期间推进的，没有先把每个失败原因拆开独立暴露。

后续需要验证：

- 首屏 `items` 从空变为有数据时能触发播放。
- 手势翻页能触发停止 / 播放。
- 搜索结果定位能触发播放。
- 程序化恢复 currentIndex 能触发播放。
- 不会重复触发同一视频的播放器创建。

### 4. PlayerController 停止逻辑改动

文件：

- [player_controller.dart](d:/Flutter/VideoPlayer/lib/features/player/controllers/player_controller.dart:231)

改动内容：

```text
stopIfCurrent / stop：
  原来先 await _disposeCurrent()
  再把 state 置为 PlayerState.initial()

  现在先把 state 置为 PlayerState.initial()
  再 await _disposeCurrent()
```

另外新增：

```text
_disposeTimeout = Duration(seconds: 2)
controller.dispose().timeout(...)
```

风险：

- “先清空业务状态再释放资源”可能是合理方向，但它是新的状态机语义。
- dispose 超时兜底会掩盖底层释放卡住的问题，可能让资源释放失败不容易暴露。
- 这部分应该有独立的 `PlayerController` 单元测试来证明：
  - stop 调用后业务状态立即清空。
  - dispose 卡住时不会阻塞业务状态。
  - dispose 正常完成时资源仍被释放。

当前这些验证还没有完整补齐。

## 已观察到的测试现象

### 第一次红灯

运行：

```text
flutter test test\integration\basic_playback_flow_test.dart
```

失败：

```text
Expected: null
Actual: 'video_001'
```

含义：

```text
滑到图片页后，播放器业务状态仍指向旧视频。
```

### 后续调试现象

添加临时日志后观察到：

```text
current item changed previous=0/video_001/video
next=1/image_001/image
```

说明 `listenManual` 已经监听到了图片项变化。

但测试仍读到旧的 `PlayerController.videoId`，进一步怀疑问题落在 `PlayerController.stop()` 的异步状态清理时机上。

### 后续超时

调整测试断言和 fake 平台后，测试出现过超时。

可能原因包括：

- fake video platform 的 stream / dispose 行为不符合 `video_player` 插件预期。
- 前面超时遗留的测试进程仍在运行。
- `PlayerController.dispose()` 等待底层释放时卡住。

该超时问题还没有被独立、干净地复现和验证。

## 后续建议

后续不要继续把所有问题混在一起改。

建议按以下顺序重新推进：

```text
1. 保留或恢复到最小红灯测试。
2. 只验证基础播放链路失败点：滑到图片页后 videoId 未清空。
3. 不在该测试中断言 disposeCount 等底层释放细节。
4. 单独为 PlayerController.stop() 写单元测试，暴露“状态清空晚于 dispose”的问题。
5. 将 PlayerController.stop() 改为“先清空业务状态，后释放底层资源”。
6. 再回到 basic_playback_flow_test.dart，验证业务链路保护是否通过。
7. 资源释放另开测试验证，不混入基础播放链路测试。
8. 最后再评估 FeedPage 是否还需要 listenManual current item 编排优化。
```

通过标准应该拆成两层：

```text
业务链路标准：
Feed 当前项是视频时播放对应视频；
Feed 当前项是非视频时 PlayerController.videoId 变为 null。

资源释放标准：
旧 VideoPlayerController 最终被 dispose；
dispose 不应长期阻塞 Feed 播放编排。
```

这两层不应该在同一个测试失败里混在一起判断。

也就是说：

```text
basic_playback_flow_test.dart 只保护业务链路：
  Feed 当前项是图片 => PlayerController.videoId 为 null
  Feed 当前项是视频 => PlayerController.videoId 为对应视频

资源释放另开 PlayerController / fake platform 测试：
  stop 后旧 VideoPlayerController 最终被 dispose
  dispose 异步细节不阻塞业务状态清空
```
