# 12.6 预加载命中热切换

本文档定义第 12.6 阶段的开发方案。

12.4 已完成真实 preload controller 生命周期，12.5 已完成单 preload 槽位的过期任务隔离和资源上限。12.6 的重点是在命中 preload 时，把已经初始化完成的 preload controller 提升为 active controller，释放真实起播收益。

## 目标

```text
当用户滑到下一条视频时，如果该视频已经预加载成功，
直接将 preloaded controller 升级为 active controller，
避免重新创建并 initialize active controller。
```

具体目标：

```text
preload hit 时复用已初始化 controller。
preload hit 时跳过 active initialize。
promote 后立即绑定 UI。
promote 后按播放意图决定是否 play。
promote 后继续刷新 current + 1 preload candidate。
preload miss 时 fallback 到现有 active 初始化流程。
记录 preload hit / miss / promoted 基础指标。
```

## 边界

本阶段只处理：

```text
单 preload controller 命中。
preloaded -> active 的资源迁移。
miss fallback。
current + 1 候选策略下的热切换。
基础 hit / miss / promoted 观测。
```

本阶段不处理：

```text
方向感知预加载。
current - 1 / current + 2 多候选。
多个 preload controller。
复杂取消队列。
平台层强取消 initialize。
首帧缓存、首帧预渲染、封面过渡。
完整 12.9 性能评估闭环。
```

命中条件：

```text
_preloadVideoId == item.id
_preloadStatus == PreloadControllerStatus.preloaded
_preloadController != null
_preloadController.value.isInitialized == true
```

以下情况都视为 miss：

```text
preloadVideoId 不匹配。
preload 仍 initializing。
preload 已 failed。
preload controller 为空。
preload controller 未 initialized。
forceRestart == true。
```

播放意图边界：

```text
preload success 不代表用户希望播放。
preload success 不能修改 wantsToPlay。
preload success 不能调用 play。
playVideo 命中 preload 后，才可以根据 playVideo 的播放意图 promote 并播放。
wantsToPlay 只能来自 playVideo 本次调用表达的播放意图。
旧 active isPlaying、preload controller 状态、preload success 都不能决定 wantsToPlay。
```

指标边界：

```text
forceRestart 是 preload bypass，不计入 preload_miss。
preload_miss 只统计本次希望使用 preload，但没有可 promote controller 的情况。
```

职责边界：

```text
PlayerController 不负责计算 current + 1 candidate。
promote 后的新 preload 调度仍由 FeedPlaybackCoordinator 在 current change 流程中完成。
```

## 执行步骤

### 1. 增加命中判断

文件：

```text
lib/features/player/controllers/player_controller.dart
```

新增私有方法：

```dart
bool _canPromotePreload(VideoFeedItem item) {
  final preloadController = _preloadController;
  return _preloadVideoId == item.id &&
      _preloadStatus == PreloadControllerStatus.preloaded &&
      preloadController != null &&
      preloadController.value.isInitialized;
}
```

`forceRestart == true` 时不走 promote。

### 2. 增加 promote 方法

新增私有方法：

```dart
Future<bool> _tryPromotePreloadToActive(
  VideoFeedItem item, {
  required bool wantsToPlay,
  PlaybackStartupSessionRef? startupSession,
})
```

promote 成功时执行：

```text
递增 _initToken，废弃旧 active 初始化任务。
递增 _preloadToken，废弃 preload 写回资格。
从 preload 槽位取出 controller / videoId / selectedQuality。
立即清空 _preloadController / _preloadVideoId / _preloadSelectedQuality / _preloadStatus。
保存旧 active controller 引用，但先不清空 active 到无 controller 状态。
把 promoted controller 赋值给 _controller。
设置 _controllerVideoId。
绑定 _controllerStartupSession。
添加 _syncFromController listener。
重置 _lastControllerIsPlaying / _lastControllerIsBuffering。
写入 PlayerState。
如果 wantsToPlay == true，记录 play requested 并调用 play。
调用 _syncFromController。
异步释放旧 active controller。
```

注意：

```text
promoted controller 不能进入 _disposePreloadControllerOnce。
promoted controller 从取出后开始只属于 active controller。
promote 时必须先让 promoted controller 接管 active，再异步释放旧 active。
不允许先清空 active / dispose 旧 active 后再绑定 promoted controller，避免 UI 短暂断开。
promote 过程中如果 active token 失效，被取出的 promoted controller 必须按临时 active controller dispose exactly once。
wantsToPlay 只能使用本次 promote 方法传入的 wantsToPlay。
```

推荐按以下机械顺序实现，避免 UI 断开、stale promote 覆盖新 active 或 dispose 路径混乱：

```text
_tryPromotePreloadToActive(item, wantsToPlay, startupSession):

1. 检查 _canPromotePreload(item)，不满足直接 return false。
2. 递增 _initToken，记录 localInitToken。
3. 递增 _preloadToken，废弃 preload 写回资格。
4. 从 preload 槽位取出 promotedController / promotedVideoId / selectedQuality。
5. 立即清空 _preloadController / _preloadVideoId / _preloadSelectedQuality / _preloadStatus。
6. 保存 oldActiveController / oldActiveVideoId。
7. 如果 localInitToken 已失效：
   - promotedController 按临时 active controller dispose once。
   - return false。
8. oldActiveController removeListener(_syncFromController)，但不等待 dispose。
9. promotedController 赋值给 _controller。
10. 写入 _controllerVideoId / _controllerStartupSession，并使用 selectedQuality 写入 PlayerState。
11. promotedController addListener(_syncFromController)。
12. 重置 _lastControllerIsPlaying / _lastControllerIsBuffering。
13. 写入 PlayerState，确保 UI 已绑定 promoted controller。
14. 调用 _syncFromController()。
15. 如果 wantsToPlay == true：
    - mark play requested。
    - 调用 promotedController.play()。
    - 再调用 _syncFromController()。
16. 异步 dispose oldActiveController。
17. return true。
```

顺序约束：

```text
第 4-5 步必须连续完成，避免取出的 controller 仍被 preload dispose 路径触达。
第 8-13 步必须保证 active 从 oldActiveController 直接切到 promotedController，中间不进入无 controller 状态。
第 16 步必须晚于 promotedController 接管 active。
如果第 15 步 play 失败，错误处理应遵循 active 播放错误语义，不回退为 preload controller。
```

### 3. 改造 playVideo 命中分支

当前 12.5 逻辑：

```text
如果 _preloadVideoId == item.id：
  disposePreload
  继续原 active 初始化
```

12.6 改为：

```text
如果 !forceRestart 且可以 promote：
  mark preload_hit
  mark preload_promoted_to_active
  promote preload -> active
  return

如果 _preloadVideoId == item.id 但不能 promote：
  mark preload_miss
  disposePreload(waitForDispose: false)
  fallback 到原 active 初始化流程

如果 forceRestart == true：
  bypass preload
  不 mark preload_miss
  按原 forceRestart active 初始化流程执行
```

原有 active 复用和新建初始化逻辑保持不变。

### 4. 保持 FeedPlaybackCoordinator 策略不变

文件：

```text
lib/features/feed/coordinators/feed_playback_coordinator.dart
```

PlayerController 不计算 next candidate。

继续保持：

```text
当前视频 playVideo。
随后调度 current + 1 preload。
```

命中后的完整链路：

```text
滑到 video_002
playVideo(video_002) promote 旧 preload
_schedulePreload(video_003)
```

不修改 `_nextVideoCandidate`。
