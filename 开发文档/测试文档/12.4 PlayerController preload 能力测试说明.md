# 12.4 PlayerController preload 能力测试说明

本文档补充第 12.4 阶段的测试范围、测试用例和回归验证命令。

12.4 的测试重点不再是“是否记录 preload 候选 id”，而是验证 `PlayerController` 内部新增的 preload 资源槽位是否满足真实生命周期隔离：

```text
preload controller 可以真实创建、initialize、持有和释放。
preload controller 不会成为 active controller。
preload 成功、失败、过期都不会污染 PlayerState。
真实 preload 后，测试中 created controller 数量会包含 active controller 和 preload controller。
```

## PlayerController 单元测试

对应文件：

```text
test/unit/player/player_controller_test.dart
```

建议覆盖：

```text
preloadVideo 会创建并 initialize 独立 controller。
preloadVideo 不调用 play。
preloadVideo 成功后 PlayerState 不变化。
preloadVideo 成功后 active videoController 仍然是原 controller。
重复 preload 同一视频不会重复创建 controller。
preload 新视频会清理旧 preload 业务槽位，并确保旧 controller 最终只释放一次。
disposePreload 会清理 preload 业务槽位，并确保旧 controller 最终只释放一次。
preload 当前 active video 会跳过。
preload initialize 失败不写 PlayerState.error。
preload initialize 失败不影响 active isPlaying / wantsToPlay。
playVideo 命中 preloadVideoId 时，12.4 会先释放 preload，再走原 active 初始化流程。
ref dispose 会释放 active controller 和 preload controller。
```

已落地用例：

```text
preloadVideo initializes an independent controller without playing:
  验证 preloadVideo 创建独立 controller、initialize 成功、状态为 preloaded。
  验证 playCount 不增加，PlayerState 完全不变化，videoController 仍为 null。

preloadVideo does not replace active playback controller:
  先播放 active 视频，再 preload 下一条。
  验证 active videoController 引用不变，PlayerState 不变，preload controller 独立存在。

preloadVideo skips current active video and disposePreload clears it:
  已有 preload 时，再请求 preload 当前 active video。
  验证旧 preload 不被当前 active 覆盖，disposePreload 后状态回到 idle。

repeated preload for the same video reuses the preload slot:
  同一 item 在 preloaded 状态下重复 preload。
  验证不重复创建 controller，不额外 dispose。

repeated preload while initializing does not create another controller:
  使用 holdInitialization 固定在 initializing。
  验证重复请求同一 item 不创建第二个 controller，release 后进入 preloaded。

preloading a new video disposes the old preload controller:
  preload A 后 preload B。
  验证 B 成为 preloadVideoId，旧 A 被释放。

preload initialize failure does not affect active playback:
  active 正在播放时让 preload initialize 失败。
  验证 preloadStatus 为 failed，preloadVideoId 保留失败目标，PlayerState.error 不变。

playVideo matching preload disposes preload before active initialization:
  preload 命中目标后调用 playVideo。
  验证 12.4 不 promote，而是清理 preload 槽位并创建新的 active controller。
```

## FeedPlaybackCoordinator 回归测试

对应文件：

```text
test/unit/feed/feed_playback_coordinator_test.dart
```

建议覆盖：

```text
handleFeedCurrentChanged 后 current + 1 视频会真实创建 preload controller。
当前项是图片时仍可以预加载后续第一个视频。
列表尾部无候选时会清理旧 preload 业务槽位。
preload 失败不影响当前 active 播放。
```

已落地用例：

```text
preloads current plus one video candidate:
  current + 1 是视频时，preloadVideoId 指向该视频。
  同时验证 hasPreloadController == true、isPreloadInitialized == true、preloadStatus == preloaded。

skips non-video items when selecting preload candidate:
  当前项或 current + 1 是图片时，继续向后找到第一个视频。
  验证不会启动 active 播放，preload 不调用 play。

clears preload candidate when no later video exists:
  列表尾部没有后续视频时调用 disposePreload。
  验证 preloadVideoId == null 且 hasPreloadController == false。
```

当前 coordinator 仍然保持 12.3 的调度策略：

```text
active 播放 / stop 流程优先。
preload 调度随后执行。
候选选择仍然只扫描 currentIndex + 1 之后的第一个视频。
```

如果后续把 `_schedulePreload` 改成 fire-and-forget，FeedPlaybackCoordinator 单测需要避免直接依赖 `handleFeedCurrentChanged` 返回时 preload 一定完成；可以改为直接测 `PlayerController.preloadVideo` 的生命周期，或在测试中等待状态进入 `preloaded / failed`。

## FakeVideoPlayerPlatform 支持

12.4 推荐 fake platform 至少记录：

```text
createdUris:
  用于区分 active 初始化和 preload 初始化都会创建 controller。

playCount:
  用于证明 preloadVideo 不调用 play。

disposeCount:
  用于证明旧 preload、新旧 active、命中 preload 后的旧资源会释放。

holdInitialization / releaseInitialization:
  用于构造 preload initializing 状态，验证重复 preload 同一视频不会重复创建。

failInitialize:
  用于验证 preload failed 不污染 active PlayerState。
```

如果要进一步覆盖“过期任务迟到”，fake platform 需要支持按 playerId 或按创建顺序分别释放 initialize / fail initialize，而不是全局一次性 release。该能力更接近 12.5 的快速滑动和过期统计，可以在后续阶段补强。

## 释放等待边界

12.4 的资源上限口径是：

```text
业务 preload 槽位最多 1 个。
平台层 VideoPlayerController 在旧 initializing 任务迟到期间可能短暂多于 1 个。
```

如果旧 preload controller 已经 `preloaded`，`disposePreload()` 可以直接释放该 controller。

如果旧 preload controller 仍在 `initializing`，`disposePreload()`、新 preload 替换旧 preload、或 `playVideo` 命中 preload 目标时，应先清理业务槽位并递增 token，使旧任务过期；旧任务后续 initialize success / failure 迟到时，再由自己的过期分支 dispose 自己创建的 controller。

测试应验证：

```text
业务槽位已经清空或切换。
旧 initializing controller 完成后最终被 dispose。
旧 controller 只会被 dispose 一次。
旧 initialize success 迟到不会把 preloadStatus 改回 preloaded。
旧 initialize failure 迟到不会把当前 preloadStatus 改为 failed。
active 起播流程不依赖旧 preload dispose 的完成时刻。
```
