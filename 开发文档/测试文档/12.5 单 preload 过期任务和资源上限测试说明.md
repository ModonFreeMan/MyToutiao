# 12.5 单 preload 过期任务和资源上限测试说明

本文档补充第 12.5 阶段的测试范围、测试用例和回归验证命令。

12.5 的测试重点不是验证 preload 是否带来起播收益，而是验证后台 preload 已经成为一个预算清晰、过期安全、不会污染 active 播放链路的能力：

```text
FeedPlaybackCoordinator 只调度 current + 1 单候选。
PlayerController 对外始终只暴露 1 个有效 preload 槽位。
旧 preload initialize 迟到后不能覆盖最新 preload 状态。
旧 preload controller 最终只 dispose 一次。
preload initialize / cleanup 异常不能泄漏到 active 链路或 Zone。
用户播放意图不受 preload 成败影响。
```

本阶段仍不实现 `preloaded controller -> active controller` 的热切换。命中 preload 时仍先清理 preload 槽位，再走原 active 初始化流程。

## PlayerController 单元测试

对应文件：

```text
test/unit/player/player_controller_test.dart
```

建议覆盖：

```text
旧 preload initializing 时，新 preload 会立即切换业务槽位。
旧 preload success 迟到不会覆盖新 preload 状态。
旧 preload failure 迟到不会覆盖新 preload 状态。
disposePreload invalidates pending initialization and disposes once。
disposePreload 在 idle 下重复调用保持幂等。
failed preload 目标再次 preload 时允许重试。
initialize throw 后会进入 failed 或 expired，并释放 controller。
provider dispose 时 pending preload 最终只释放一次。
preload 当前 active video 会跳过，且不清理无关 preload。
playVideo 命中 preloadVideoId 时会先清理 preload，再走原 active 初始化。
用户暂停后 preload success 不会修改 wantsToPlay。
preload success / failure 不写 PlayerState.error。
preload success / failure 不调用 play。
preload cleanup / stale dispose 异常不会向外泄漏。
```

已落地用例：

```text
late stale preload initialization success does not override current preload:
  preload A hold 后启动 preload B。
  先 release A，验证 B 仍是当前 preloadVideoId，状态仍为 initializing。
  再 release B，验证 B 进入 preloaded。
  验证旧 A 被 dispose 一次。

late stale preload initialization failure does not override current preload:
  preload A hold 后启动 preload B。
  先 fail A，验证不会把 B 的状态改成 failed。
  再 release B，验证 B 正常进入 preloaded。

disposePreload invalidates pending initialization and disposes once:
  preload A initializing 时调用 disposePreload。
  验证业务槽位立即变为 idle，且不会同步 dispose pending controller。
  release A 后，验证 A 迟到分支最终 dispose 一次。

disposePreload is idempotent when idle:
  idle 状态下重复调用 disposePreload。
  验证状态保持 idle，不创建额外 dispose。

failed preload target can be retried:
  第一次 preload initialize 失败后，preloadStatus 为 failed 且保留失败目标。
  同一目标再次 preload 时重新创建 controller，并可进入 preloaded。

preload initialize failure swallows cleanup dispose errors:
  initialize 失败后让 fake platform 的 dispose 抛错。
  验证 preloadVideo Future 仍 completes。
  验证失败被转成 preloadStatus.failed，不污染 PlayerState。

stale preload completion swallows cleanup dispose errors:
  preload A 过期后，A 的 stale completion 触发 cleanup dispose。
  让 cleanup dispose 抛错。
  验证 A 的 Future 仍 completes，B 仍是当前有效 preload 槽位。

provider dispose invalidates pending preload and disposes it once:
  provider dispose 时，如果 preload 仍 initializing，只清业务槽位并使任务过期。
  release pending initialize 后，旧 controller 由迟到任务 dispose 一次。

preload completion does not restore paused playback intent:
  active 播放后用户主动 pause，使 wantsToPlay == false。
  preload success 后，验证 wantsToPlay 仍为 false，isPlaying 仍为 false，playCount 不增加。
```

## FeedPlaybackCoordinator 单元测试

对应文件：

```text
test/unit/feed/feed_playback_coordinator_test.dart
```

12.5 后 `_schedulePreload` 已改成 fire-and-forget：

```text
有候选时：
  unawaited(playerController.preloadVideo(candidate))

无候选时：
  await playerController.disposePreload()
```

因此 coordinator 测试不能再假设 `handleFeedCurrentChanged` 返回时 preload 一定已经完成。测试应直接等待 fake platform release initialize，或等待状态进入 `preloaded / failed / idle`。

已落地用例：

```text
preloads current plus one video candidate:
  current + 1 是视频时，preloadVideoId 指向该视频。
  等待后台 preload 完成后，验证 hasPreloadController、isPreloadInitialized 和 preloadStatus。

skips non-video items when selecting preload candidate:
  当前项或 current + 1 是图片时，继续向后寻找第一个视频。
  验证图片项不会触发 active 播放，preload 不调用 play。

clears preload candidate when no later video exists:
  列表尾部没有后续视频时调用 disposePreload。
  验证 preloadVideoId == null 且 hasPreloadController == false。

slow preload initialization does not block current feed change:
  hold 所有初始化。
  handleFeedCurrentChanged 先进入 active 初始化。
  release active 后，handleFeedCurrentChanged 可以返回，而 preload 仍保持 initializing。
  验证后台 preload 不阻塞当前 active 播放链路。

rapid current changes keep only latest preload candidate visible:
  模拟 current index 快速从 0 切到 1。
  preload A 仍 initializing 时启动 preload B。
  先 release A，验证 A 迟到不会覆盖 B。
  再 release B，验证最终 preload 指向最新 current + 1 候选。
```

## FakeVideoPlayerPlatform 支持

12.5 测试依赖 fake platform 提供更细的异步控制能力。

通用 fake helper：

```text
test/helpers/fake_video_player_platform.dart
```

当前支持：

```text
createdUris:
  记录 controller 创建顺序和数据源。

disposeCount:
  验证旧 preload / active controller 最终释放次数。

holdInitialization:
  阻止 initialize 自动完成，用于构造 pending preload。

releaseInitializationForCreation(index):
  按创建顺序释放指定 controller 的 initialize。

failInitializationForCreation(index):
  按创建顺序让指定 controller initialize 失败。

failInitialize:
  让所有 initialize 失败，用于 failed preload 重试测试。

failDispose:
  让 dispose 抛错，用于验证 preload cleanup 异常不会泄漏。

playCount:
  验证 preload success / failure 不调用 play。
```

FeedPlaybackCoordinator 测试内的本地 fake platform 也增加了按创建顺序 release initialize 的能力，用于稳定构造：

```text
active A initialize hold。
preload B initialize hold。
先 release active，再保持 preload pending。
快速切换后先 release 旧 preload，再 release 新 preload。
```

## 异常边界

12.5 对 preload 异常的要求是：

```text
preload initialize 失败不能影响 active controller。
preloadVideo 对外 Future 不应因为后台 initialize / cleanup / stale completion 抛出未捕获异常。
FeedPlaybackCoordinator 不负责 catch preloadVideo 异常。
```

当前实现通过 `PlayerController._disposePreloadControllerOnce` 统一封住 preload cleanup 异常：

```text
waitForDispose == true:
  try / catch dispose Future。

waitForDispose == false:
  unawaited(dispose.catchError(...))。
```

注意：该异常吞吐只作用于 preload 专用释放路径，不改变 active controller dispose 的错误传播语义。

## 集成测试回归

12.5 不改变用户可见的播放、暂停、清晰度切换和横屏语义，但后台 preload 已经变成 fire-and-forget，因此集成测试关注点是：

```text
首屏 active 播放仍正常。
滑到图片页后 active 会停止。
滑到视频页后 active 会切换到对应视频。
搜索结果、清晰度切换不受后台 preload 影响。
preload miss 时仍回退到原 active 初始化流程。
```
