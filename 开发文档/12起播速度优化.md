# 第 12 阶段：Feed 视频预加载优化

本阶段目标聚焦 Feed 视频预加载能力：先建立起播链路的可观测指标，再逐步引入 current + 1 预加载 MVP、预加载命中热切换、过期任务隔离、方向感知候选选择和资源释放策略。

首帧呈现、封面过渡、首帧缓存和预渲染不放在本阶段实现，独立进入第 13 阶段。

核心原则：

```text
先可观测
再整理生命周期
再轻量预加载
再命中热切换
最后用 p50 / p90 / p95 验证收益
```

优化重点关注 Feed 翻页后的视频初始化等待、预加载命中率和实际起播等待，尤其关注 p90 / p95 长尾体验，而不是只看平均值。

## 12.1 起播性能指标埋点

目标：

```text
先知道当前起播慢在哪里，并为预加载前后对比建立 baseline。
本阶段只增加观测能力，不改变播放逻辑。
```

需要记录的关键时间点：

```text
feed_item_visible_at
controller_initialize_start
controller_initialize_end
first_frame_rendered_at
play_requested_at
actual_playing_at
buffering_start
buffering_end
preload_hit
```

核心指标：

```text
起播耗时 = actual_playing_at - play_requested_at
初始化耗时 = controller_initialize_end - controller_initialize_start
预加载命中率 = preload_hit / visible_items
```

输出要求：

```text
baseline 下的起播耗时 p50 / p90 / p95
baseline 下的初始化耗时 p50 / p90 / p95
起播后前 N 秒 buffering 次数和总时长
```

通过标准：

```text
不改变现有播放行为。
可以稳定采集起播相关时间点。
可以输出当前版本的 baseline 数据。
指标采集失败不能影响正常播放。
```

## 12.2 FeedPlaybackCoordinator 生命周期整理

目标：

```text
在实现预加载前，先明确播放协调层的资源调度边界。
预加载调度决策应收口到 FeedPlaybackCoordinator，而不是分散到单个 Feed 卡片里。
底层 VideoPlayerController 的创建、复用和释放仍由 PlayerController 执行。
```

职责划分：

```text
FeedItemWidget:
  负责内容 UI 展示和用户交互入口。
  不直接决定当前播放项、预加载项或播放器释放策略。

VideoFeedCard:
  负责绑定 PlayerState 并展示视频画面、播放控件、进度、清晰度和横屏入口。
  可以把点击、横屏等交互转发给 FeedPlaybackCoordinator。
  不负责全局 active / preload / dispose 调度。

FeedPlaybackCoordinator:
  负责跨模块播放业务编排和资源调度决策。
  决定谁应该成为 active 目标、谁应该成为 preload 候选。
  多 controller 缓存阶段再决定哪些项离开保留范围需要释放。
  调用 PlayerController 执行播放、暂停、恢复、预加载、热切换和释放。
  不直接持有或创建底层 VideoPlayerController。

PlayerController:
  负责播放器状态机和底层播放器实例生命周期。
  创建、初始化、播放、暂停、seek、清晰度切换、复用预加载实例、释放不再保留的 controller。
  对外提供 active / preload / dispose 执行能力和结果状态。

Observability:
  负责旁路记录 active / preload / dispose 链路中的观测事件和 session 归属。
  可以记录初始化、首帧、播放确认、缓冲、预加载命中 / 失败等事件。
  不参与播放调度决策，不直接控制播放器生命周期。

Repository / DataSource:
  负责提供视频地址、封面、清晰度信息。
  不参与播放调度和播放器生命周期管理。
```

需要明确的调度语义：

```text
active controller:
  业务上当前应该展示或播放的视频。
  由 FeedPlaybackCoordinator 决策，底层 controller 由 PlayerController 持有和执行。

preloaded controller:
  下一条候选视频已经完成预加载或正在预加载。
  FeedPlaybackCoordinator 决定候选项，PlayerController 管理真实预加载实例。

disposed controller:
  离开保留范围、需要释放的播放器资源。
  单 preload 阶段只释放当前 preload。
  多 controller 缓存阶段再由 FeedPlaybackCoordinator 决定释放范围，PlayerController 执行 dispose。

failed controller:
  初始化或预加载失败的资源状态。
  失败结果不应污染当前 active 播放状态，也不应修改用户播放意图。
```

状态建议：

```text
idle
initializing
preloaded
active
disposing
failed
```

这些状态用于表达调度层和执行层之间的资源状态，不表示 Feed Widget 可以直接管理播放器资源。

推荐调用关系：

```text
FeedPage / VideoFeedCard
  -> FeedPlaybackCoordinator
  -> PlayerController
  -> VideoPlayerController
```

调度边界：

```text
FeedPlaybackCoordinator 决定：
  active target
  preload target
  单 preload 阶段的 preload 清理
  多 controller 缓存阶段的 dispose targets
  用户播放意图是否应该保持
  失败结果是否应该忽略或 fallback

PlayerController 执行：
  创建 controller
  初始化 controller
  播放 / 暂停 / 恢复
  切换清晰度
  promote preloaded -> active
  dispose controller

Observability 观察：
  feed visible
  initialize start / end / failed
  first frame rendered
  play requested / actual playing
  buffering start / end
  preload hit / miss / failed / disposed
```

通过标准：

```text
Feed Widget 不直接负责全局播放器资源调度。
FeedPlaybackCoordinator 可以表达当前播放项和预加载项的生命周期。
FeedPlaybackCoordinator 不直接持有底层 VideoPlayerController。
PlayerController 继续负责真实播放器实例生命周期。
现有播放、暂停、横屏、质量切换链路职责不被打散。
```

## 12.3 current + 1 候选选择 MVP

详细设计：

```text
开发文档/12.3 current + 1 候选选择.md
```

目标：

```text
只选择当前项之后的第 1 个视频作为 preload candidate。
如果 current + 1 是图片或图文，则继续向后寻找第 1 个视频。
暂不处理 current - 1、current + 2 和方向回滑。
```

基本策略：

```text
current index 变化时：
1. 保留当前 active controller。
2. 如果当前项是视频，则启动当前视频的正常 active 播放流程。
3. 在 items 中向后查找第 1 个 VideoFeedItem。
4. 如果存在候选视频，则请求 PlayerController 记录该 preload 目标。
5. 如果不存在候选视频，则清空当前 preload 候选。
```

PlayerController 在 12.3 只提供轻量调度结果：

```text
preloadVideo(item):
  如果 item 已经是 active video，跳过。
  否则只记录 _preloadVideoId = item.id。
  不创建 preload controller。
  不 initialize。
  不 play。
  不修改 PlayerState。

disposePreload():
  清空 _preloadVideoId。
```

通过标准：

```text
只产生 0 或 1 个 preload candidate。
非视频 item 不会占用预加载名额。
预加载候选不改变当前 active videoId。
预加载候选不修改 wantsToPlay。
预加载候选可通过 PlayerController.preloadVideoId 验证。
找不到候选时会清空旧 preloadVideoId。
```

## 12.4 PlayerController preload 能力

目标：

```text
给 PlayerController 增加独立的 preload controller 管理能力。
预加载只完成 controller 创建和 initialize，不 play、不绑定为 active。
```

基本能力：

```text
preloadVideo(item):
  如果 item 已经是 active video，跳过。
  如果 item 已经是当前 preload video，跳过。
  如果已有旧 preload，取消或标记过期，并释放旧 preload controller。
  创建并 initialize 新 preload controller。
  initialize 成功后记录为 preloaded。
  initialize 失败只记录失败，不影响 active controller。

disposePreload():
  释放当前 preload controller。
```

通过标准：

```text
active controller 和 preload controller 相互独立。
preload initialize 失败不改变当前播放状态。
preload 完成不会自动播放。
preload 完成不会触发 UI 切换。
重复预加载同一视频不会重复创建 controller。
```

## 12.5 单 preload 过期任务和资源上限

目标：

```text
确保快速滑动时，旧 preload 完成后不会污染当前预加载状态。
第一版只允许 active + preload 两类 controller 存活。
```

策略：

```text
active controller 最多 1 个。
preload controller 最多 1 个。
同一时间只有 1 个 preload initialize 任务拥有写回 preload 状态的资格。
过期 initializing 任务可以短暂存在，但不属于有效业务槽位。
controller 总数默认不超过 2 个。
当前 active 初始化优先级高于任何后台 preload。
预加载失败不影响当前 active controller。
快速滑动时，过期 preload 完成后必须被忽略。
过期 preload 完成后只能 dispose-once 自己创建的 controller。
preloadVideo 不允许向外泄漏未捕获的后台异常。
预加载只代表资源准备，不代表用户希望播放。
```

需要明确区分：

```text
isInitialized
isPreloaded
isVisible
wantsToPlay
isPlaying
```

其中：

```text
wantsToPlay 表示业务播放意图。
预加载完成不能直接修改用户的播放意图。
```

通过标准：

```text
当前视频播放不受后台预加载影响。
后台 preload 不会阻塞 active 初始化关键路径。
预加载 miss 时仍回退到原初始化流程。
用户主动暂停后不会被预加载逻辑误恢复。
快速连续滑动不会让旧 preload 覆盖最新 preload。
离开保留范围的 preload 会被释放。
```

## 12.6 预加载命中热切换

目标：

```text
当用户滑到下一条视频时，如果该视频已经预加载成功，
直接将 preloaded controller 升级为 active controller。
```

命中逻辑：

```text
如果当前可见视频命中 preload：
  preloaded controller -> active controller
  立即绑定 UI
  根据 wantsToPlay 决定是否播放
  12.6 阶段继续刷新 current + 1 video candidate
  12.7 阶段按最近滑动方向刷新单 preload candidate

如果没有命中 preload：
  fallback 到原来的初始化流程
```

需要记录：

```text
preload_hit
preload_miss
preload_promoted_to_active
preload_disposed_before_hit
```

通过标准：

```text
preload hit 时可以复用已准备好的 controller。
preload miss 时行为与现状一致。
热切换不覆盖暂停意图。
热切换不破坏横屏返回、质量切换、Feed 覆盖恢复等既有链路。
```

## 12.7 方向感知预加载扩展

目标：

```text
在 current + 1 链路、单 preload 生命周期和命中热切换稳定后，扩展方向感知候选选择。
本阶段只改变“预加载谁”，不改变 PlayerController 的 preload 生命周期。
默认仍保持单 preload controller，不引入多槽位、不引入 preload queue。
扩展重点是提高单 preload candidate 的命中率，同时避免后台初始化拖慢当前 active 起播。
```

候选优先级：

```text
最近方向向下：
  1. 当前项之后的第 1 个视频
  2. 当前项之后的第 2 个视频
  3. 当前项之前的第 1 个视频

最近方向向上：
  1. 当前项之前的第 1 个视频
  2. 当前项之后的第 1 个视频

方向未知或首屏：
  1. 当前项之后的第 1 个视频
```

方向 fallback：

```text
forward:
  先找后方视频。
  后方没有视频时，fallback 到前方视频。

backward:
  先找前方视频。
  前方没有视频时，fallback 到后方视频。

unknown:
  只找后方视频。
  后方没有视频时返回空。
  unknown 不 fallback 到前方视频。
```

阶段边界：

```text
FeedPlaybackCoordinator 记录最近 current index 和滑动方向。
FeedPlaybackCoordinator 内部通过 Feed item id 前缀判断 items 是否非追加式替换。
分页 loadMore 只是在尾部追加 items，不重置方向。
刷新、搜索结果切换、筛选变化或重新加载造成 items 非追加式替换时，重置方向状态。
FeedPage 不承担方向状态或 items 换源判断。
PlayerController 继续只负责真实 preload controller 生命周期、过期隔离和 promote。
MVP 仍只保留 1 个有效 preload controller。
多 warm / preload slot 留到后续阶段，不属于 12.7 默认交付范围。
```

## 12.8 快速滑动取消和多候选 dispose 策略，暂缓

目标：

```text
避免快速滑动时大量过期初始化任务污染当前播放状态，
并控制播放器资源占用。
12.5 只处理单 preload controller 的过期和释放。
12.7 只处理方向感知单候选。
本节在此基础上扩展到多候选队列、更多 warm / preload slot 和更完整的 dispose 策略。
```

策略：

```text
每次 preload 绑定 generation / token。
当 current index 改变时，旧 preload 标记为过期。
过期 preload 即使初始化完成，也不能升级为 active。
远离视口的 controller 进入 dispose。
dispose 失败或延迟不能阻塞当前 active 初始化。
```

需要关注：

```text
Android / iOS 解码器数量限制。
多个 controller 同时 initialize 可能造成卡顿。
快速滑动时预加载队列可能频繁重排。
旧 controller dispose 不应阻塞新 controller 创建。
```

通过标准：

```text
快速连续滑动不会让旧视频突然开始播放。
旧 preload 完成后不会覆盖当前 videoId。
资源释放策略可控，不出现 controller 无限制增长。
当前播放链路优先级高于后台预加载。
```

## 12.9 预加载效果评估与回归验证

目标：

```text
用数据说明预加载优化效果，并确保既有播放链路没有回退。
```

对照组：

```text
baseline:
  当前未开启预加载的版本。

preload only:
  12.6 对照组开启 current + 1 video 预加载和热切换。
  12.7 对照组开启方向感知单 preload candidate 和热切换。
  后续扩展对照组再开启多候选 / 多 warm slot。
```

需要输出：

```text
起播耗时 p50 / p90 / p95
初始化耗时 p50 / p90 / p95
预加载命中率
预加载候选数和平均存活 controller 数
buffering 次数和总时长
预加载失败率
快速滑动下的过期 preload 数量
```

回归场景：

```text
Feed 首屏自动播放
上下滑切换视频和图文内容
用户主动暂停后恢复
Feed 被搜索页覆盖后返回
清晰度切换
横屏播放和返回竖屏
快速连续滑动
预加载失败 fallback
```

通过标准：

```text
preload only 相比 baseline 在 p90 / p95 起播耗时上有明确收益。
预加载命中率可以解释优化效果。
active 初始化耗时不能因为后台预加载明显恶化。
buffering、崩溃率、内存占用没有明显恶化。
所有既有播放器、Feed、横屏和清晰度测试保持通过。
flutter analyze 无新增问题。
```

## 阶段结论

第 12 阶段的核心不是堆叠“预加载 + 预渲染”功能，而是建立一条可验证的起播优化闭环：

```text
先用指标定位瓶颈；
再用 current + 1 验证预加载链路；
再用方向感知单 preload candidate 提升命中率；
后续再根据数据决定是否扩展为多候选、多 warm slot 和更复杂的 dispose 策略；
再通过 preload hit 热切换释放收益；
最后用数据决定是否进入第 13 阶段首帧呈现优化。
```

预加载和播放意图必须保持解耦：

```text
预加载只表示资源准备完成，
不表示用户希望播放。
```

最终验收应以数据为准，重点关注 p90 / p95 长尾体验，而不是仅凭单次手感判断优化是否成功。
