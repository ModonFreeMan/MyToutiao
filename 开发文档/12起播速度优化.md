# 第 12 阶段：起播速度优化

本阶段目标不是一次性引入复杂预渲染，而是先建立 Feed 起播链路的可观测指标，再基于数据逐步引入下一条视频预加载、预加载命中热切换和首帧呈现优化。

核心原则：

```text
先可观测
再整理生命周期
再轻量预加载
再命中热切换
再考虑首帧优化
最后用 p50 / p90 / p95 验证收益
```

优化重点关注 Feed 翻页后的视频首帧等待、初始化等待和实际起播等待，尤其关注 p90 / p95 长尾体验，而不是只看平均值。

## 12.1 起播性能指标埋点

目标：

```text
先知道当前起播慢在哪里。
本阶段只增加观测能力，不改变播放逻辑。
```

需要记录的关键时间点：

```text
feed_item_visible_at
controller_initialize_start
controller_initialize_end
first_frame_rendered_at
play_requested_at
actual_playing_at
buffering_start
buffering_end
preload_hit
```

核心指标：

```text
首帧耗时 = first_frame_rendered_at - feed_item_visible_at
起播耗时 = actual_playing_at - play_requested_at
初始化耗时 = controller_initialize_end - controller_initialize_start
预加载命中率 = preload_hit / visible_items
```

输出要求：

```text
baseline 下的首帧耗时 p50 / p90 / p95
baseline 下的起播耗时 p50 / p90 / p95
baseline 下的初始化耗时 p50 / p90 / p95
起播后前 N 秒 buffering 次数和总时长
```

通过标准：

```text
不改变现有播放行为。
可以稳定采集起播相关时间点。
可以输出当前版本的 baseline 数据。
指标采集失败不能影响正常播放。
```

## 12.2 FeedPlaybackCoordinator 生命周期整理

目标：

```text
在实现预加载前，先明确播放协调层的资源调度边界。
预加载逻辑应放在 FeedPlaybackCoordinator，而不是分散到单个 Feed 卡片里。
```

职责划分：

```text
FeedItemWidget:
  负责展示内容 UI。

VideoFeedCard:
  负责绑定播放器状态和展示播放控件。

FeedPlaybackCoordinator:
  负责 active / preload / dispose 的调度。

Repository / DataSource:
  负责提供视频地址、封面、清晰度信息。
```

需要明确的 controller 生命周期：

```text
active controller:
  当前正在展示或播放的视频 controller。

preloaded controller:
  下一条候选视频已经初始化完成或正在初始化的 controller。

disposed controller:
  离开保留范围、需要释放的 controller。

failed controller:
  初始化或预加载失败的 controller，不应污染当前播放状态。
```

状态建议：

```text
idle
initializing
preloaded
active
disposing
failed
```

通过标准：

```text
Feed Widget 不直接负责全局播放器资源调度。
FeedPlaybackCoordinator 可以表达当前播放项和预加载项的生命周期。
现有播放、暂停、横屏、质量切换链路职责不被打散。
```

## 12.3 下一条视频预加载

目标：

```text
只做 current + 1 的轻量预加载。
不扩大到 current + 2 或 current - 1，避免资源压力过早失控。
```

基本策略：

```text
current index 变化时：
1. 保留当前 active controller。
2. 预加载 current + 1。
3. 取消或忽略旧的 preload 任务。
4. dispose 远离视口的 controller。
```

资源约束：

```text
同一时间只允许有限数量 controller 存活。
预加载失败不影响当前 active controller。
快速滑动时，过期 preload 完成后必须被忽略。
预加载只代表资源准备，不代表用户希望播放。
```

需要明确区分：

```text
isInitialized
isPreloaded
isVisible
wantsToPlay
isPlaying
```

其中：

```text
wantsToPlay 表示业务播放意图。
预加载完成不能直接修改用户的播放意图。
```

通过标准：

```text
当前视频播放不受下一条预加载影响。
只预加载 current + 1。
预加载 miss 时仍回退到原初始化流程。
用户主动暂停后不会被预加载逻辑误恢复。
```

## 12.4 预加载命中热切换

目标：

```text
当用户滑到下一条视频时，如果该视频已经预加载成功，
直接将 preloaded controller 升级为 active controller。
```

命中逻辑：

```text
如果下一条命中 preload：
  preloaded controller -> active controller
  立即绑定 UI
  根据 wantsToPlay 决定是否播放
  再预加载新的 current + 1

如果没有命中 preload：
  fallback 到原来的初始化流程
```

需要记录：

```text
preload_hit
preload_miss
preload_promoted_to_active
preload_disposed_before_hit
```

通过标准：

```text
preload hit 时可以复用已准备好的 controller。
preload miss 时行为与现状一致。
热切换不覆盖暂停意图。
热切换不破坏横屏返回、质量切换、Feed 覆盖恢复等既有链路。
```

## 12.5 快速滑动取消、过期任务忽略与 dispose 策略

目标：

```text
避免快速滑动时大量过期初始化任务污染当前播放状态，
并控制播放器资源占用。
```

策略：

```text
每次 preload 绑定 generation / token。
当 current index 改变时，旧 preload 标记为过期。
过期 preload 即使初始化完成，也不能升级为 active。
远离视口的 controller 进入 dispose。
dispose 失败或延迟不能阻塞当前 active 初始化。
```

需要关注：

```text
Android / iOS 解码器数量限制。
多个 controller 同时 initialize 可能造成卡顿。
旧 controller dispose 不应阻塞新 controller 创建。
```

通过标准：

```text
快速连续滑动不会让旧视频突然开始播放。
旧 preload 完成后不会覆盖当前 videoId。
资源释放策略可控，不出现 controller 无限制增长。
当前播放链路优先级高于后台预加载。
```

## 12.6 首帧呈现优化

目标：

```text
在数据证明首帧等待仍是主要瓶颈时，再继续优化首帧呈现。
优先做低风险方案，暂不引入复杂离屏渲染池。
```

优先级：

```text
第一档：
  使用已初始化 controller 尽快显示首帧，减少黑屏。

第二档：
  如果首帧仍慢，再考虑封面图或首帧缓存图过渡。

暂不优先做：
  离屏渲染池
  多 controller 预渲染
  复杂首帧缓存系统
```

通过标准：

```text
首帧优化不改变播放意图。
首帧占位或封面过渡不遮挡正常播放控件。
不同平台上表现可接受。
如果数据表明收益有限，可以停止在轻量方案。
```

## 12.7 效果评估与回归验证

目标：

```text
用数据说明优化效果，并确保既有播放链路没有回退。
```

对照组：

```text
baseline:
  当前未开启预加载和首帧优化的版本。

preload only:
  只开启 current + 1 预加载和热切换。

preload + first frame optimization:
  在预加载基础上开启首帧呈现优化。
```

需要输出：

```text
起播耗时 p50 / p90 / p95
首帧耗时 p50 / p90 / p95
初始化耗时 p50 / p90 / p95
预加载命中率
buffering 次数和总时长
预加载失败率
快速滑动下的过期 preload 数量
```

回归场景：

```text
Feed 首屏自动播放
上下滑切换视频和图文内容
用户主动暂停后恢复
Feed 被搜索页覆盖后返回
清晰度切换
横屏播放和返回竖屏
快速连续滑动
预加载失败 fallback
```

通过标准：

```text
preload only 相比 baseline 在 p90 / p95 起播耗时上有明确收益。
预加载命中率可以解释优化效果。
buffering、崩溃率、内存占用没有明显恶化。
所有既有播放器、Feed、横屏和清晰度测试保持通过。
flutter analyze 无新增问题。
```

## 阶段结论

第 12 阶段的核心不是堆叠“预加载 + 预渲染”功能，而是建立一条可验证的起播优化闭环：

```text
先用指标定位瓶颈；
再用 current + 1 预加载降低初始化等待；
再通过 preload hit 热切换释放收益；
最后根据数据决定是否继续做首帧呈现优化。
```

预加载和播放意图必须保持解耦：

```text
预加载只表示资源准备完成，
不表示用户希望播放。
```

最终验收应以数据为准，重点关注 p90 / p95 长尾体验，而不是仅凭单次手感判断优化是否成功。
