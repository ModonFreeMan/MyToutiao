# 第 13 阶段：首帧呈现优化

本阶段在第 12 阶段预加载能力稳定后启动，目标是优化用户看到视频画面的等待体验。

第 13 阶段不负责播放器资源预加载、不负责 preload controller 生命周期，也不改变播放意图语义。它只处理首帧呈现、封面过渡和必要时的首帧缓存策略。

## 核心原则

```text
先复用第 12 阶段已有数据
再判断首帧是否仍是主要瓶颈
先做低风险呈现优化
再考虑缓存或预渲染
最后用 p50 / p90 / p95 验证收益
```

## 13.1 首帧瓶颈确认

目标：

```text
确认预加载优化后，用户等待主要是否仍来自首帧呈现。
如果预加载已经解决主要长尾问题，本阶段可以暂停。
```

需要参考：

```text
first_frame_ms p50 / p90 / p95
startup_ms p50 / p90 / p95
initialize_ms p50 / p90 / p95
preload_hit_rate
buffering 次数和总时长
```

通过标准：

```text
能够说明首帧等待是否仍有独立优化价值。
不会仅凭单次手感进入复杂首帧方案。
```

## 13.2 已初始化 controller 首帧呈现

目标：

```text
当 controller 已初始化时，尽快让 VideoPlayer widget/texture 进入渲染树。
减少已准备好但 UI 仍显示黑屏或封面的时间。
```

约束：

```text
不改变 wantsToPlay。
不自动播放用户已暂停的视频。
不影响横屏返回、质量切换、Feed 覆盖恢复。
```

通过标准：

```text
已初始化视频能更快进入可见渲染状态。
播放控件、进度条和横屏入口不被遮挡。
现有播放器测试保持通过。
```

## 13.3 封面过渡

目标：

```text
如果首帧仍慢，用视频封面作为过渡层减少黑屏感。
封面只作为视觉过渡，不伪造播放成功状态。
```

策略：

```text
未初始化或等待首帧时显示封面。
controller 已初始化并完成首帧呈现后淡出封面。
播放失败时展示错误态，而不是停留在封面假正常状态。
```

通过标准：

```text
封面过渡不遮挡正常播放控件。
封面过渡不会让用户误以为视频已经播放。
暂停态、失败态、buffering 态语义清晰。
```

## 13.4 首帧缓存，暂缓

目标：

```text
只有当封面过渡仍不足以改善体验时，才考虑首帧缓存。
默认不在第一版实现复杂缓存系统。
```

暂缓内容：

```text
离屏渲染池
多 controller 预渲染
跨会话首帧图片缓存
复杂磁盘缓存和淘汰策略
```

进入条件：

```text
预加载命中率较高，但 first_frame_ms p90 / p95 仍明显偏高。
封面过渡无法满足体验要求。
设备资源和内存占用风险可控。
```

## 13.5 效果评估与回归验证

对照组：

```text
baseline:
  第 12 阶段预加载优化后的版本。

first frame display:
  已初始化 controller 尽快呈现首帧。

cover transition:
  在首帧等待期间显示封面过渡。
```

需要输出：

```text
首帧耗时 p50 / p90 / p95
起播耗时 p50 / p90 / p95
封面显示时长 p50 / p90 / p95
buffering 次数和总时长
播放失败率
```

回归场景：

```text
Feed 首屏自动播放
上下滑切换视频和图文内容
用户主动暂停后恢复
Feed 被搜索页覆盖后返回
清晰度切换
横屏播放和返回竖屏
快速连续滑动
预加载命中和 miss
播放失败 fallback
```

通过标准：

```text
first_frame_ms p90 / p95 有明确改善。
startup_ms 不明显恶化。
封面或首帧过渡不改变播放意图。
所有既有播放器、Feed、横屏和清晰度测试保持通过。
flutter analyze 无新增问题。
```

## 阶段结论

第 13 阶段只解决“用户何时看到画面”的问题，不解决“资源何时准备好”的问题。

资源准备属于第 12 阶段预加载优化；画面呈现属于第 13 阶段首帧优化。
