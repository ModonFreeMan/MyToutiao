# 12.1 起播性能指标埋点设计说明

本设计用于建立 Feed 起播链路的可观测能力，明确当前版本起播慢在哪里。

本阶段只增加观测能力，不改变播放逻辑、不引入预加载、不引入首帧优化。

## 核心原则

12.1 采集的是业务侧近似观测指标，不是播放器底层真实物理指标。

这些指标用于稳定对比同一业务链路在不同版本下的变化，例如 baseline、预加载后、首帧优化后。它们不用于证明底层播放器真实解码首帧、Surface 首帧或音视频稳定播放时间。

设计约束：

```text
只做内存记录。
不做 I/O。
不做网络请求。
不 await。
不触发 UI rebuild。
不参与播放决策。
采集失败不能影响正常播放。
所有异步回调必须绑定 session 并由 metrics 内部校验归属。
```

## 指标语义

### feed_item_visible_at

含义：

```text
Feed 当前项切换到该视频的时间。
```

注意：

```text
它不是严格的屏幕可见时间。
它不保证该视频画面已经真实出现在屏幕上。
```

记录位置：

```text
FeedPlaybackCoordinator.handleFeedCurrentChanged(index)
```

当当前 Feed item 是 `VideoFeedItem` 时创建新的 startup session。

### controller_initialize_start

含义：

```text
业务侧开始创建并初始化 VideoPlayerController 的时间。
```

记录位置：

```text
PlayerController.playVideo()
PlayerController.switchQuality()
```

在调用 `VideoPlayerController.networkUrl` 和 `initialize()` 前记录。

### controller_initialize_end

含义：

```text
明确归属于原 session 的 controller initialize 成功完成时间。
```

注意：

```text
initialize_ms 的样本不要求 session 仍然 active。
只要 initialize 成功事件能够明确归属于原 session，就计入初始化耗时。
```

快速滑动后，原 session 可能已经 expired，但其 initialize 成功回调仍可作为初始化耗时样本。

### first_frame_rendered_at

含义：

```text
Flutter 层首次把目标视频的 VideoPlayer widget/texture 放入渲染树后的 post-frame 时间。
```

注意：

```text
它是 Flutter 层近似首帧。
它不是播放器底层真实首帧。
它不是解码首帧。
它不是 Surface/Texture 真实像素输出时间。
```

记录位置：

```text
VideoPlayerView
```

当当前视频满足以下条件后，通过 `WidgetsBinding.instance.addPostFrameCallback` 记录：

```text
playerState.videoId == item.id
playerState.isInitialized == true
controller != null
!playerState.isLandscapeRendering
该 session 尚未记录 first_frame_rendered_at
```

post-frame 回调触发时必须再次由 metrics 内部校验 session 归属。

### play_requested_at

含义：

```text
业务侧请求播放某个视频的时间。
```

记录位置：

```text
PlayerController.playVideo()
PlayerController.resume()
PlayerController.togglePlayPause()
PlayerController.ensurePlaybackIntent()
```

12.1 重点关注 Feed 起播 baseline。本阶段覆盖 `playVideo()` 中自动起播和复用 controller 起播路径，同时记录 resume/toggle/ensurePlaybackIntent 触发的播放请求，确保后续 report 能区分不同请求来源。

12.1 的 `play_requested_at` 只保留 session 内首次播放请求时间，用于计算 `startup_ms`。后续重复播放请求不覆盖首次时间点，但需要累计到 `play_request_sources`，用于区分请求来源。

### actual_playing_at

含义：

```text
video_player 的 value.isPlaying 首次变为 true 的时间。
```

注意：

```text
它是业务侧播放状态确认。
它可能早于真实画面稳定播放。
它不代表音视频已经稳定输出。
```

记录位置：

```text
PlayerController._syncFromController()
```

通过监听 `isPlaying` 的状态边沿记录：

```text
false -> true
```

### buffering_start / buffering_end

含义：

```text
video_player 的 value.isBuffering 边沿变化时间。
```

记录位置：

```text
PlayerController._syncFromController()
```

通过监听 `isBuffering` 的状态边沿记录：

```text
false -> true: buffering_start
true -> false: buffering_end
```

统计口径只覆盖 `actual_playing_at` 后前 N 秒。

### preload_hit

12.1 不实现预加载，因此 baseline report 中不输出 `0%`。

必须输出：

```text
preload_enabled: false
preload_hit_rate: null
preload_hit_rate_label: "N/A"
```

避免误解成“预加载命中率差”。

## Session 模型

所有埋点必须绑定 startup session。

不能只按 `videoId` 归属样本，因为同一个视频可能被多次切入视口，快速滑动时也可能出现迟到回调。

数据模型：

```dart
class PlaybackStartupSession {
  final int sessionId;
  final String videoId;
  final int feedIndex;
  final DateTime createdAt;

  PlaybackStartupSessionStatus status;

  DateTime? feedItemVisibleAt;
  DateTime? controllerInitializeStart;
  DateTime? controllerInitializeEnd;
  DateTime? firstFrameRenderedAt;
  DateTime? playRequestedAt;
  DateTime? actualPlayingAt;

  bool preloadHit;
  Object? error;

  final List<PlaybackBufferingSpan> bufferingSpans;
}
```

session 状态机至少包含：

```text
active
completed
expired
failed
closed
```

状态含义：

```text
active:
  当前 Feed 起播观测 session。

completed:
  已采集到本次起播所需关键时间点。

expired:
  Feed 已切换到其它 session，当前 session 不再接收首帧、playing、buffering 等 active-only 事件。

failed:
  初始化或播放链路失败，不能进入成功样本。

closed:
  播放停止、Feed 切到非视频、页面销毁或 metrics 被手动关闭。
```

## markXXX 归属校验

调用方不得先判断 `isCurrentSession`。

所有 `markXXX` 方法必须在 metrics 内部统一做 session 归属校验，避免调用方漏校验。

接口设计：

```dart
PlaybackStartupSessionRef markFeedItemVisible({
  required String videoId,
  required int feedIndex,
});

void markPlayRequested(PlaybackStartupSessionRef session);
void markControllerInitializeStart(PlaybackStartupSessionRef session);
void markControllerInitializeEnd(PlaybackStartupSessionRef session);
void markControllerInitializeFailed(PlaybackStartupSessionRef session, Object error);
void markFirstFrameRendered(PlaybackStartupSessionRef session);
void markActualPlaying(PlaybackStartupSessionRef session);
void markBufferingStart(PlaybackStartupSessionRef session);
void markBufferingEnd(PlaybackStartupSessionRef session);
void markSessionExpired(PlaybackStartupSessionRef session);
void markSessionClosed(PlaybackStartupSessionRef session);
PlaybackStartupBaselineReport buildReport();
```

内部校验规则：

```text
sessionId 必须存在。
videoId 必须匹配。
closed / failed session 不再接收成功事件。
first_frame_rendered_at / actual_playing_at / buffering_start / buffering_end 必须属于当前 active session。
controller_initialize_end 只要求明确归属于原 initialize session，不要求仍 active。
迟到事件不抛错，计入 ignored_late_events。
重复事件不覆盖首次时间点，并计入 ignored_late_events。
```

所有 `markXXX` 必须同步返回，不能 `await`。

所有 `markXXX` 必须吞掉内部异常：

```dart
void _safeRecord(void Function() action) {
  try {
    action();
  } catch (_) {
    // Metrics must never affect playback.
  }
}
```

## 异步过期处理

过期不只发生在 `initialize()`。

以下回调都可能迟到，必须通过 session 归属校验：

```text
initialize success
initialize failure
VideoPlayer listener 中的 isPlaying 变化
VideoPlayer listener 中的 isBuffering 变化
VideoPlayerView postFrameCallback
Feed autoplay grace period 之后的 playVideo
横屏返回后的 ensurePlaybackIntent 间接播放状态变化
```

当 Feed 当前项切到新视频时：

```text
旧 active session 标记为 expired。
新视频创建新的 active session。
```

当 Feed 当前项切到非视频时：

```text
旧 active session 标记为 closed。
```

## 核心指标

首帧耗时：

```text
first_frame_ms = first_frame_rendered_at - feed_item_visible_at
```

起播耗时：

```text
startup_ms = actual_playing_at - play_requested_at
```

初始化耗时：

```text
initialize_ms = controller_initialize_end - controller_initialize_start
```

起播后前 N 秒 buffering：

```text
startup_buffering_count
startup_buffering_total_ms
```

预加载命中率：

```text
12.1: N/A
12.3/12.4 后: preload_hit / preload_visible_items
```

## 样本过滤规则

### first_frame_ms

有效样本必须满足：

```text
feed_item_visible_at 存在
first_frame_rendered_at 存在
session 未 failed / closed
first_frame_rendered_at >= feed_item_visible_at
```

expired session 的 first frame 事件不进入样本，因为 post-frame 迟到时已经无法代表当前可见视频。

### startup_ms

有效样本必须满足：

```text
play_requested_at 存在
actual_playing_at 存在
session 未 failed / closed
actual_playing_at >= play_requested_at
```

`actual_playing_at` 必须来自当前 active session 的 `isPlaying` 边沿。

### initialize_ms

有效样本必须满足：

```text
controller_initialize_start 存在
controller_initialize_end 存在
initialize 成功
controller_initialize_end >= controller_initialize_start
事件可明确归属于原 initialize session
```

注意：

```text
initialize_ms 不要求 session 仍 active。
同一 session 内重复 initialize 事件不覆盖首次 initialize 时间点。
```

这是为了保留快速滑动下真实发生过的初始化成本，帮助判断初始化本身是否是瓶颈。

12.1 的 initialize 样本是 session 级首次初始化成本。清晰度切换会接入 initialize start/end，但若该 session 已经记录过首次初始化时间点，则后续 initialize 事件按重复事件处理，不覆盖首次样本。后续如果需要独立统计清晰度切换初始化耗时，应扩展为多 initialize span。

### buffering

有效样本必须满足：

```text
actual_playing_at 存在
buffering span 与 actual_playing_at 后 N 秒窗口有交集
```

窗口：

```text
[actual_playing_at, actual_playing_at + bufferingWindow]
```

未闭合 buffering span 的截断规则写死：

```text
effectiveEnd = min(reportAt, actualPlayingAt + bufferingWindow)
```

如果已闭合，则：

```text
effectiveEnd = min(bufferingEnd, reportAt, actualPlayingAt + bufferingWindow)
```

只统计窗口内的交集时长。

## 分位数口径

baseline 需要输出：

```text
first_frame_ms p50 / p90 / p95
startup_ms p50 / p90 / p95
initialize_ms p50 / p90 / p95
startup_buffering_count p50 / p90 / p95
startup_buffering_total_ms p50 / p90 / p95
```

分位数使用 nearest-rank：

```text
index = ceil(sampleCount * percentile) - 1
```

样本为空时输出 `null`，并通过样本数说明不足。

## 模块设计

新增模块：

```text
lib/features/observability/metrics/playback_startup_metrics.dart
lib/features/observability/metrics/playback_startup_session.dart
lib/features/observability/metrics/playback_startup_report.dart
lib/features/observability/providers/observability_provider.dart
```

新增测试：

```text
test/unit/observability/playback_startup_metrics_test.dart
```

`PlaybackStartupMetrics` 不得继承 `ChangeNotifier`，也不得作为会频繁刷新 UI 的 Riverpod state。

通过普通 `Provider<PlaybackStartupMetrics>` 提供单例实例，但 `markXXX` 不能更新任何参与 UI build 的状态。

`PlaybackStartupMetrics` 属于 Observability 模块，是播放链路的旁路观测能力。
它可以被 `FeedPlaybackCoordinator`、`PlayerController`、`VideoPlayerView` 在关键事件点调用，
但不能参与播放调度决策，也不能直接控制播放器生命周期。

## 测试设计

单元测试覆盖：

```text
完整 session 输出 first_frame_ms / startup_ms / initialize_ms。
markXXX 内部会忽略不属于当前 active session 的 first frame / playing / buffering。
expired session 的 initialize success 仍可计入 initialize_ms。
failed initialize 不进入 initialize_ms，并计入 initialize_failed_sessions。
重复事件不会覆盖首次时间点。
未闭合 buffering span 按 min(reportAt, actualPlayingAt + N 秒) 截断。
preload 未启用时 hit rate 为 null / N/A。
空样本分位数输出 null。
metrics 内部异常不会向调用方抛出。
```

回归测试关注：

```text
Feed 首屏自动播放。
上下滑切换视频和图文内容。
用户主动暂停后恢复。
Feed 被搜索页覆盖后返回。
清晰度切换。
横屏播放和返回竖屏。
快速连续滑动。
初始化失败 fallback。
```

## 验收标准

```text
不改变现有播放行为。
稳定采集业务侧近似起播相关时间点。
所有埋点以 session 为单位归属。
所有 markXXX 内部完成 session 校验。
所有异步回调迟到时不会串样本。
baseline report 输出 p50 / p90 / p95。
baseline report 包含有效样本数和丢弃样本数。
preload 未启用时显示 N/A。
指标采集失败不能影响正常播放。
flutter analyze 无新增问题。
flutter test 无回归。
```

## 阶段结论

12.1 的价值不是得到绝对真实的播放器底层时间，而是建立一把语义清晰、归属正确、可重复对比的业务侧尺子。

后续 12.3 / 12.4 引入预加载和热切换后，才能用同一套口径回答：

```text
初始化等待是否下降？
起播等待是否下降？
首帧近似等待是否下降？
p90 / p95 长尾是否改善？
buffering 是否恶化？
预加载命中率能否解释收益？
```
