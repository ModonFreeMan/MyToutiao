# 12.9 预加载效果评估与回归验证

本文档定义第 12.9 阶段的开发方案。

12.6 已完成单 preload 命中热切换，12.7 已完成方向感知单候选，12.8 已完成快速滑动下的 preload 调度节流。12.9 不继续扩展预加载策略，而是用同一套业务侧指标验证优化是否真的有效，并把播放器主链路回归风险收住。

## 目标

```text
用数据对比 baseline 与 preload only 的起播体验。
验证 p50 / p90 / p95 是否改善，重点看 p90 / p95 长尾。
解释收益是否来自 preload 命中和 promote。
确认后台 preload 没有明显拖慢 active 初始化。
确认 buffering、失败率、资源占用和既有播放链路没有回退。
形成可重复执行的评估与回归流程。
```

本阶段要回答的问题：

```text
preload only 相比 baseline，startup_ms 是否下降？
initialize_ms 是否没有明显恶化？
preload_hit_rate 是否能解释 startup_ms 的改善？
快速滑动下过期 preload 是否被节流控制住？
用户暂停、横屏、清晰度切换、Feed 覆盖返回是否仍然稳定？
```

## 边界

本阶段只处理：

```text
补齐评估所需的 report 字段。
增加 debug-only 的 report 导出入口。
定义 baseline / preload only 的采样流程。
定义数据对比口径。
补充 metrics 和回归测试。
记录一次 12.9 自动化验证结果。
```

本阶段不处理：

```text
不新增多 preload controller。
不新增 preload queue。
不新增 current + 2 / current - 1 同时预加载。
不做平台层强取消 initialize。
不做内存 profiler 或性能平台接入。
不做线上埋点、网络上报或文件自动上传。
不做首帧缓存、预渲染或封面过渡。
不改变 PlayerController 的 preload / promote 状态机。
不改变 FeedPlaybackCoordinator 的方向感知和 debounce 策略。
```

12.9 是验证闭环，不是新功能堆叠阶段。

## 评估对象

对照组：

```text
baseline:
  关闭 preload 调度或使用未开启 preload 的版本。
  preload_enabled 应为 false，preload_hit_rate_label 为 N/A。

preload only:
  开启 12.6 preload hit promote。
  开启 12.7 方向感知单候选。
  开启 12.8 preload 调度稳定窗口。
  仍然只允许 1 个有效 preload controller。
```

不在本阶段比较：

```text
多候选 / 多 slot。
首帧缓存 / 预渲染。
不同清晰度 preload 策略。
不同 debounce 时长调参矩阵。
```


如缺字段，只补最小字段，不重写 metrics 结构。

字段语义保持：

```text
preload_visible_items = preload_hit 或 preload_miss 的 session 数。
preload_hit_rate = preload_hits / preload_visible_items。
forceRestart bypass 不计入 preload_miss。
无 preload 样本时 preload_hit_rate 为 null，label 为 N/A。
```

### 2. 增加 debug-only 导出入口

目标：

```text
可以手动拿到当前进程内 PlaybackStartupMetrics.buildReport().toJson()。
导出入口不参与播放决策。
导出失败不影响播放。
```

建议选择最小实现之一：

```text
方案 A:
  增加 debug-only service/helper，返回 Map<String, Object?>。

方案 B:
  在已有调试入口中增加打印 JSON 的方法。

方案 C:
  测试和手工验证阶段直接通过 provider 读取 buildReport，不增加 UI。
```

推荐先选方案 C 或 A，避免为了 12.9 新增页面、按钮或复杂 UI。

约束：

```text
不做自动文件写入。
不做网络上报。
不在 markXXX 中 await。
不让 report 导出触发 UI rebuild。
```

### 3. 定义采样流程

同一套 Feed 数据、同一设备或模拟器、同一网络条件下分别采集：

```text
baseline:
  至少 30 个 video visible 样本。

preload only:
  至少 30 个 video visible 样本。
```

建议手工路径：

```text
1. 冷启动进入 Feed。
2. 等首屏视频进入播放。
3. 匀速向下滑动一轮，覆盖视频和图文混排。
4. 向上回滑一轮，覆盖方向感知候选。
5. 快速连续滑动一轮，覆盖 debounce 和过期 preload。
6. 停在一个视频，主动暂停，再继续滑动。
7. 进入搜索覆盖页后返回 Feed。
8. 执行一次清晰度切换。
9. 进入横屏播放并返回竖屏。
10. 导出 report JSON。
```

采样记录需要同时保存：

```text
代码版本或 commit。
运行模式：baseline / preload only。
设备或模拟器信息。
Feed 数据来源。
样本数量。
是否发生初始化失败或明显网络波动。
report JSON。
```

### 4. 数据对比口径

对比时先看样本有效性：

```text
valid_startup_samples 是否足够。
valid_initialize_samples 是否足够。
incomplete_sessions 是否异常偏高。
ignored_late_events 是否异常偏高。
initialize_failed_sessions 是否异常偏高。
```

再看核心收益：

```text
startup_ms p90 / p95:
  preload only 应低于 baseline。

preload_hit_rate:
  应能解释 startup_ms 改善。
  如果 hit rate 低但 startup 改善明显，需要复查口径。

initialize_ms p90 / p95:
  preload only 不应比 baseline 明显恶化。

startup_buffering_count / startup_buffering_total_ms:
  preload only 不应明显恶化。
```

建议使用克制的验收阈值：

```text
preload only 的 startup_ms p90 或 p95 有可见下降。
preload_hit_rate 大于 0，且 promoted 数与 hit 数一致或接近。
initialize_ms p90 / p95 没有明显上升。
buffering p90 / p95 没有明显上升。
没有新增崩溃、未捕获异常或 controller 泄漏迹象。
```

本阶段不设硬编码百分比门槛。样本量、网络和测试设备波动较大时，以“可解释、可重复”为优先。

## 验收标准

```text
report 可以稳定输出 12.9 必需字段。
baseline 无 preload 样本时显示 N/A，而不是 0%。
preload only 可以输出 preload_visible_items / hits / misses / promoted / hit rate。
preload hit rate 口径与 12.6 forceRestart bypass 规则一致。
startup_ms / initialize_ms / first_frame_ms / buffering 分位数仍使用 12.1 口径。
指标采集和 report 导出不改变播放行为。
指标采集失败不会影响播放。
手工采样完成后，preload only 相比 baseline 在 startup_ms p90 或 p95 上有可解释收益。
initialize_ms p90 / p95 没有明显恶化。
buffering 次数和总时长没有明显恶化。
快速滑动下不会启动已取消的 pending preload。
旧 preload 迟到不会覆盖当前 preload 状态。
active controller 和有效 preload controller 仍保持单槽位约束。
用户暂停意图不被 preload success 或 promote 覆盖。
清晰度切换、横屏、Feed 覆盖返回无回归。
flutter analyze 无新增问题。

最终 flutter test 通过。
```

## 阶段结论

12.9 的重点是让第 12 阶段收口：

```text
不是继续增加 preload 能力，
而是证明当前单 preload + 方向感知 + 快速滑动节流是否值得保留。
```

如果 p90 / p95 起播收益明确，且回归稳定，可以进入第 13 阶段首帧呈现优化。

如果收益不明显，应优先复查命中率、样本口径和 active 初始化长尾，再决定是否进入多候选或多 slot，而不是直接扩大 preload 复杂度。
