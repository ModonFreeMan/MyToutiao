# 12.7 方向感知预加载扩展

本文档定义第 12.7 阶段的开发方案。

12.6 已完成单 preload controller 命中后的热切换能力：当前可见视频命中已完成初始化的 preload controller 时，可以直接 promote 为 active controller。12.7 在此基础上只扩展候选选择策略，让 Feed 预加载不再固定依赖 current + 1，而是根据最近滑动方向选择更可能命中的视频。

本阶段仍保持单 preload controller，不引入多槽位、不引入 preload queue，也不改变 PlayerController 的 preload 生命周期。

## 目标

```text
根据最近 Feed 滑动方向选择 preload candidate。
默认仍只保留 1 个 preload controller。
active 初始化优先级继续高于后台 preload。
preload success 不改变播放意图。
preload promote 仍复用 12.6 playVideo 命中逻辑。
```

具体目标：

```text
FeedPlaybackCoordinator 记录最近 current index 和滑动方向。
forward 时优先预加载后方视频。
backward 时优先预加载前方视频。
unknown / 首屏时保持 current + 1 语义。
非视频 item 不占用预加载名额。
无候选时只释放 preload。
Feed 列表刷新或换源时重置方向状态。
```

## 边界

本阶段只处理：

```text
方向感知候选选择。
单 preload controller 调度。
候选为空时清理 preload。
命中后的 promote 继续复用 12.6 playVideo 逻辑。
```

本阶段不处理：

```text
多个 preload controller。
preload queue 并发初始化。
current + 2 和 current - 1 同时预加载。
平台层强取消 initialize。
复杂 dispose 队列。
首帧缓存、首帧预渲染、封面过渡。
完整 12.9 性能评估闭环。
```

职责边界：

```text
FeedPlaybackCoordinator:
  判断最近滑动方向。
  按方向选择 1 个 preload candidate。
  调用 PlayerController.preloadVideo 或 disposePreload。

PlayerController:
  继续负责真实 preload controller 生命周期。
  继续负责 preload hit promote。
  不负责计算方向。
  不负责扫描 Feed items。
```

播放意图边界：

```text
方向感知只影响 preload candidate。
preload success 不能修改 wantsToPlay。
preload success 不能调用 play。
只有 playVideo 当前目标命中 preload 时，才能 promote。
```

资源边界：

```text
active controller 最多 1 个。
preload controller 最多 1 个。
业务 controller 总数仍为 active + preload。
快速滑动时旧 initializing preload 仍按 12.5 过期任务语义处理。
```

Feed 数据边界：

```text
方向状态只在同一批 Feed items 内有效。
搜索、筛选、刷新、重新加载 Feed 数据时必须重置方向状态。
新列表 currentIndex 不能和旧列表 currentIndex 比较方向。
旧列表 currentIndex = 20，新列表 currentIndex = 0 时不能判断为 backward。
```

## 执行步骤

### 1. 增加方向状态

文件：

```text
lib/features/feed/coordinators/feed_playback_coordinator.dart
```

新增方向枚举：

```dart
enum FeedScrollDirection {
  unknown,
  forward,
  backward,
}
```

新增字段：

```dart
int? _lastCurrentIndex;
FeedScrollDirection _lastScrollDirection = FeedScrollDirection.unknown;
List<String> _lastFeedItemIds = const <String>[];
```

新增方法：

```dart
void _resetScrollDirection() {
  _lastCurrentIndex = null;
  _lastScrollDirection = FeedScrollDirection.unknown;
}

void _resetScrollDirectionIfFeedItemsReplaced(List<FeedItem> items) {
  final itemIds = items.map((item) => item.id).toList(growable: false);
  if (_isSameFeedItemPrefix(_lastFeedItemIds, itemIds)) {
    _lastFeedItemIds = itemIds;
    return;
  }

  _lastFeedItemIds = itemIds;
  _resetScrollDirection();
}

bool _isSameFeedItemPrefix(List<String> previous, List<String> next) {
  if (previous.isEmpty) {
    return true;
  }

  if (next.length < previous.length) {
    return false;
  }

  for (var index = 0; index < previous.length; index++) {
    if (previous[index] != next[index]) {
      return false;
    }
  }

  return true;
}

FeedScrollDirection _updateScrollDirection(int currentIndex) {
  final lastCurrentIndex = _lastCurrentIndex;
  _lastCurrentIndex = currentIndex;

  if (lastCurrentIndex == null) {
    _lastScrollDirection = FeedScrollDirection.unknown;
    return _lastScrollDirection;
  }

  if (currentIndex > lastCurrentIndex) {
    _lastScrollDirection = FeedScrollDirection.forward;
  } else if (currentIndex < lastCurrentIndex) {
    _lastScrollDirection = FeedScrollDirection.backward;
  }

  return _lastScrollDirection;
}
```

调用约束：

```text
handleFeedCurrentChanged 读取 feedState.items 后，先调用 _resetScrollDirectionIfFeedItemsReplaced。
再调用 _updateScrollDirection 更新方向。
同 index 重复触发时保持旧方向。
分页 loadMore 只是在尾部追加 items，不重置方向。
Feed 列表刷新、搜索结果切换、筛选变化或重新加载造成 items 非追加式替换时，Coordinator 内部重置方向。
```

不强制本阶段新增 Feed generation 字段。Coordinator 通过 Feed items id 前缀判断是否为同一批数据：

```text
previous 为空：
  视为首次加载，不额外 reset。

next 以前一批 item ids 为前缀：
  视为分页追加，不 reset。

next 不是以前一批 item ids 为前缀：
  视为刷新 / 搜索结果切换 / 筛选变化 / 重新加载导致的换源，reset 方向。
```

这样 reset 逻辑仍收口在 FeedPlaybackCoordinator，FeedPage 只负责监听当前项变化并通知协调器，不承担方向状态或数据换源判断。

### 2. 实现单候选选择

将 current + 1 单向选择替换为方向感知单候选：

```dart
VideoFeedItem? _preloadCandidate(
  List<FeedItem> items,
  int currentIndex, {
  required FeedScrollDirection direction,
})
```

建议只保留两个扫描 helper：

```dart
VideoFeedItem? _nextVideoAfter(List<FeedItem> items, int currentIndex)
VideoFeedItem? _nextVideoBefore(List<FeedItem> items, int currentIndex)
```

扫描规则：

```text
_nextVideoAfter 从 currentIndex + 1 向后扫描。
_nextVideoBefore 从 currentIndex - 1 向前扫描。
只返回 VideoFeedItem。
非视频 item 直接跳过。
index 越界时不能抛错。
items 为空时返回 null。
```

方向 fallback 表必须写死：

```text
forward:
  先找后方视频。
  后方没有视频时，fallback 到前方视频。

backward:
  先找前方视频。
  前方没有视频时，fallback 到后方视频。

unknown:
  只找后方视频。
  后方没有视频时返回空。
  unknown 不 fallback 到前方视频。
```

对应伪代码：

```dart
VideoFeedItem? _preloadCandidate(
  List<FeedItem> items,
  int currentIndex, {
  required FeedScrollDirection direction,
}) {
  return switch (direction) {
    FeedScrollDirection.forward =>
      _nextVideoAfter(items, currentIndex) ??
      _nextVideoBefore(items, currentIndex),
    FeedScrollDirection.backward =>
      _nextVideoBefore(items, currentIndex) ??
      _nextVideoAfter(items, currentIndex),
    FeedScrollDirection.unknown => _nextVideoAfter(items, currentIndex),
  };
}
```

### 3. 调整调度入口

`handleFeedCurrentChanged` 中：

```text
先读取 feedState。
先让 FeedPlaybackCoordinator 内部判断 items 是否换源，必要时重置方向。
再更新方向。
再计算当前 item。
再计算 preload candidate。
当前 active play / stop 流程保持原语义。
active 流程后调度 preload。
```

调度方法保持单候选：

```dart
Future<void> _schedulePreload(VideoFeedItem? candidate) async {
  final playerController = _ref.read(playerControllerProvider.notifier);
  if (candidate == null) {
    await playerController.disposePreload();
    return;
  }

  unawaited(playerController.preloadVideo(candidate));
}
```

重复候选处理：

```text
FeedPlaybackCoordinator 不判断 candidate 是否等于当前 preloadVideoId。
重复候选由 PlayerController.preloadVideo 内部跳过。
failed preload 目标是否重试也沿用 PlayerController 既有语义。
```

无候选处理：

```text
candidate == null 时只调用 disposePreload。
disposePreload 只能清理 preload。
disposePreload 不得 stop active。
disposePreload 不得清空 PlayerState.videoId。
disposePreload 不得修改 wantsToPlay。
当前项是视频但没有 preload candidate 时，active playVideo 仍应正常执行。
```

### 4. 保持 PlayerController 不变

本阶段不修改：

```text
PlayerController.preloadVideo
PlayerController.disposePreload
PlayerController._canPromotePreload
PlayerController._tryPromotePreloadToActive
PreloadControllerStatus
```

12.7 只改变“预加载谁”，不改变“如何预加载”和“如何 promote”。

## 验收标准

```text
首屏或方向未知时，候选选择等价于 current + 1。
Feed 列表刷新或换源后方向状态会重置为 unknown。
items 换源检测在 FeedPlaybackCoordinator 内部完成，不由 FeedPage 判断。
分页追加 items 不会重置方向。
旧列表高 index 切到新列表低 index 时，不会被误判为 backward。
向下滑动时，优先选择当前项之后的第 1 个视频。
向下滑动且后方无视频时，fallback 到当前项之前的第 1 个视频。
向上滑动时，优先选择当前项之前的第 1 个视频。
向上滑动且前方无视频时，fallback 到当前项之后的第 1 个视频。
方向未知时只查找后方视频，后方无视频时返回空。
方向未知时即使前方有视频，也不会 fallback 到前方视频。
非视频 item 不占用预加载名额。
index 越界时不会抛错。
没有候选时只清理 preload，不影响 active。
重复候选由 PlayerController.preloadVideo 跳过或重试。
仍然只存在 1 个有效 preload controller。
preload success 不修改 PlayerState。
preload success 不修改 wantsToPlay。
preload success 不调用 play。
preload hit promote 仍复用 12.6 逻辑。
现有播放、暂停、清晰度切换、横屏和 Feed 覆盖恢复链路无回归。
flutter analyze 无新增问题。
相关 flutter test 无回归。
```

## 阶段结论

12.7 的核心是提高单 preload candidate 的命中概率。

本阶段只做方向感知单槽位：

```text
方向判断在 FeedPlaybackCoordinator。
候选仍然只有 1 个。
真实 preload 生命周期仍在 PlayerController。
命中 promote 仍复用 12.6。
```

多槽位、更多候选和复杂 dispose 策略留到后续阶段处理。
