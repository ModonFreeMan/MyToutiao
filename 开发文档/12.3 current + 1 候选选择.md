# 12.3 current + 1 候选选择 MVP

本文档定义第 12.3 阶段的 Feed 预加载候选选择方案。

本阶段只实现 current + 1 方向上的单候选调度，不实现真实 preload controller 生命周期，不创建后台 `VideoPlayerController`，不做 initialize，不做命中热切换。

12.3 的目标是先让播放协调层稳定回答：

```text
当前 Feed index 变化后，下一条应该预加载的视频是谁？
这个候选是否会污染当前 active 播放状态？
PlayerController 后续 12.4 应该从哪个入口接入真实 preload 能力？
```

## 目标

```text
只选择当前项之后的第 1 个视频作为 preload candidate。
如果 current + 1 是图片或图文，则继续向后寻找第 1 个视频。
暂不处理 current - 1、current + 2 和方向回滑。
```

本阶段通过 `PlayerController.preloadVideo(item)` 记录轻量 preload 调度结果：

```text
记录 preload 目标 videoId。
不创建 preload controller。
不 initialize。
不 play。
不修改 PlayerState。
```

## 非目标

```text
不实现真实 preload controller 生命周期。
不实现 active controller 与 preload controller 的实例隔离。
不实现 preload 命中后的 promote / hot switch。
不处理快速滑动下的 generation / token 过期隔离。
不实现多候选队列。
不记录 preload hit / miss 指标。
```

这些能力进入后续阶段：

```text
12.4 PlayerController preload 能力
12.5 单 preload 过期任务和资源上限
12.6 预加载命中热切换
```

## 基本策略

`currentIndex` 变化时：

```text
1. 保留当前 active controller 的既有语义。
2. 如果当前项是视频，则启动当前视频的正常 active 播放流程。
3. 如果当前项不是视频，则保持现有停止 active 播放流程。
4. 在 items 中从 currentIndex + 1 开始向后查找第 1 个 VideoFeedItem。
5. 如果存在候选视频，则请求 PlayerController 记录该 preload 目标。
6. 如果不存在候选视频，则清理当前 preload 候选。
```

调度顺序建议：

```text
active 播放 / 停止流程优先。
preload candidate 调度随后执行。
```

原因：

```text
当前可见项永远是 active 目标。
preload 只是后台候选，不应抢占 active 播放链路语义。
```

## FeedPlaybackCoordinator 设计

候选选择逻辑收口到 `FeedPlaybackCoordinator`。

建议新增私有方法：

```dart
VideoFeedItem? _nextVideoCandidate(List<FeedItem> items, int currentIndex) {
  for (var index = currentIndex + 1; index < items.length; index++) {
    final item = items[index];
    if (item case final VideoFeedItem videoItem) {
      return videoItem;
    }
  }

  return null;
}
```

`handleFeedCurrentChanged(index)` 中先读取候选：

```dart
final feedState = _ref.read(feedViewModelProvider);
final item = _itemAt(feedState.items, index);
final preloadCandidate = _nextVideoCandidate(feedState.items, index);
```

当前项是视频时：

```dart
await playerController.playVideo(
  videoItem,
  startupSession: startupSession,
);
await _schedulePreload(preloadCandidate);
```

当前项不是视频时：

```dart
await playerController.stop();
await _schedulePreload(preloadCandidate);
```

推荐增加调度辅助方法：

```dart
Future<void> _schedulePreload(VideoFeedItem? candidate) async {
  final playerController = _ref.read(playerControllerProvider.notifier);
  if (candidate == null) {
    await playerController.disposePreload();
    return;
  }

  await playerController.preloadVideo(candidate);
}
```

如果不希望等待轻量记录，也可以使用 `unawaited`，但 12.3 阶段该方法不做耗时工作，直接 `await` 更利于测试和顺序表达。

## PlayerController 轻量 preload 入口

12.3 不建议让 `preloadVideo(...)` 完全空实现。

完全空实现只能证明：

```text
FeedPlaybackCoordinator 找到了候选。
FeedPlaybackCoordinator 调用了 preloadVideo。
```

但无法证明：

```text
preload 目标是谁。
preload 是否污染 active 状态。
后续 12.4 要接哪个入口。
```

因此 12.3 至少保留一个轻量调度结果：

```dart
String? _preloadVideoId;

String? get preloadVideoId => _preloadVideoId;

Future<void> preloadVideo(VideoFeedItem item) async {
  if (state.videoId == item.id) {
    return;
  }

  _preloadVideoId = item.id;
}

Future<void> disposePreload() async {
  _preloadVideoId = null;
}
```

语义：

```text
_preloadVideoId 表示当前调度层选中的 preload 目标。
它不是已初始化 controller。
它不是 preloaded 状态。
它不代表用户希望播放。
```

`disposePreload()` 在 12.3 中只清理 `_preloadVideoId`。
12.4 中可以在同一入口扩展为释放真实 preload controller。

## 状态隔离要求

`preloadVideo(item)` 不得修改：

```text
PlayerState.videoId
PlayerState.wantsToPlay
PlayerState.isPlaying
PlayerState.isInitializing
PlayerState.isInitialized
PlayerState.currentPosition
PlayerState.duration
PlayerState.error
PlayerState.isLandscapeRendering
```

当候选视频等于当前 active 视频时：

```text
preloadVideo 直接跳过。
不覆盖 _preloadVideoId。
不修改 PlayerState。
```

当后续不存在候选视频时：

```text
FeedPlaybackCoordinator 调用 disposePreload。
PlayerController 清空 _preloadVideoId。
```

这样可以避免滑到列表尾部时保留旧候选，导致调度语义误判。

## 与现有播放链路的关系

当前项是视频：

```text
FeedPlaybackCoordinator
  -> PlayerController.playVideo(currentVideo)
  -> PlayerController.preloadVideo(nextVideoCandidate)
```

当前项是图片或图文：

```text
FeedPlaybackCoordinator
  -> PlayerController.stop()
  -> PlayerController.preloadVideo(nextVideoCandidate)
```

后续没有候选视频：

```text
FeedPlaybackCoordinator
  -> PlayerController.disposePreload()
```

预加载候选只描述“下一条可能要准备的视频”。
它不改变当前 Feed 可见项，也不改变当前 active 播放器。

## 测试设计

建议覆盖 `FeedPlaybackCoordinator` 与 `PlayerController` 两层。

### FeedPlaybackCoordinator 测试

测试内容：

```text
current + 1 是视频时，preloadVideoId 等于 current + 1 视频 id。
current + 1 是图片，后续第 1 个视频成为 preloadVideoId。
当前项是图片时，仍然向后寻找第 1 个视频作为 preload candidate。
后续没有视频时，preloadVideoId 被清空。
每次 current index 变化最多产生 0 或 1 个 preload candidate。
非视频 item 不占用 preload 名额。
```

通过标准：

```text
候选选择只从 currentIndex + 1 开始向后扫描。
扫描遇到第 1 个 VideoFeedItem 后立即停止。
不会选择 current - 1、current + 2 的第二候选或方向回滑候选。
```

### PlayerController 测试

测试内容：

```text
preloadVideo 记录 preloadVideoId。
preloadVideo 遇到当前 active video 时直接跳过。
disposePreload 清空 preloadVideoId。
preloadVideo 不修改当前 active videoId。
preloadVideo 不修改 wantsToPlay。
preloadVideo 不触发 isPlaying。
preloadVideo 不创建或替换当前 videoController。
```

通过标准：

```text
preload 调度结果可被测试读取。
active 播放状态不受 preload 调度影响。
用户播放意图不受 preload 调度影响。
```

## 验收标准

```text
只产生 0 或 1 个 preload candidate。
非视频 item 不会占用预加载名额。
预加载候选不改变当前 active videoId。
预加载候选不修改 wantsToPlay。
预加载候选可通过 PlayerController.preloadVideoId 验证。
找不到候选时会清空旧 preloadVideoId。
preloadVideo / disposePreload 成为 12.4 真实 preload controller 生命周期的稳定入口。
flutter analyze 无新增问题。
相关 flutter test 无回归。
```

## 阶段结论

12.3 的重点不是提前实现预加载资源管理，而是把“选谁预加载”这件事做实。

本阶段完成后，系统应具备一个稳定、可测试、不会污染 active 播放状态的 current + 1 preload 调度结果：

```text
FeedPlaybackCoordinator 决定候选。
PlayerController 记录候选。
PlayerState 不被候选污染。
真实 preload controller 生命周期留给 12.4。
```
