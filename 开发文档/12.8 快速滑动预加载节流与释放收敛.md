# 12.8 快速滑动预加载节流与释放收敛

本文档定义第 12.8 阶段的开发方案。

12.7 已完成方向感知单 preload candidate 选择。12.8 不扩展多 preload controller，不引入 preload queue，也不改变 PlayerController 的 promote 逻辑。本阶段只在单 preload 基础上处理快速滑动时的无效预加载调度，让后台 preload 更克制、更容易验证。

## 目标

```text
快速滑动时减少短生命周期 preload 初始化。
保持 active 起播优先级高于后台 preload。
继续只保留 1 个有效 preload controller。
不引入多 slot。
不引入 preload queue。
不改变 12.6 preload hit promote 语义。
不改变 12.7 方向感知候选选择语义。
无候选、换源、页面离开时清理 preload 更明确。
```

核心目标：

```text
active 播放立即执行。
preload 调度延迟一个很短的稳定窗口。
稳定窗口内 current index 再变化时，取消上一轮尚未启动的 preload。
只有最新一轮候选在窗口结束后才调用 PlayerController.preloadVideo。
candidate == null 时立即 disposePreload，不等待稳定窗口。
Feed items 非追加式换源时取消 pending preload，并清理旧 preload。
```

建议默认稳定窗口：

```text
120ms
```

这个窗口只用于抑制快速滑动中的瞬时候选，不参与播放、不影响 active 初始化。

## 边界

本阶段只处理：

```text
FeedPlaybackCoordinator 内部的 preload 调度节流。
取消尚未启动的 pending preload。
无候选时立即清理 preload。
Feed 换源时取消 pending preload 并清理旧 preload。
快速滑动下 active playVideo 不被 preload 等待拖慢。
快速滑动下旧 initializing preload 迟到仍不能覆盖当前状态。
```

本阶段不处理：

```text
多个 preload controller。
preload queue。
current + 1 和 current - 1 同时预加载。
current + 2 预加载。
平台层强取消 VideoPlayerController.initialize。
复杂 dispose 队列。
按内存、机型、网络动态调整 preload slot。
首帧缓存、首帧预渲染、封面过渡。
完整 12.9 性能评估闭环。
```

职责边界：

```text
FeedPlaybackCoordinator:
  继续负责方向感知候选选择。
  负责 preload 调度节流。
  负责取消尚未启动的 pending preload。
  candidate 为空或换源时调用 disposePreload。

PlayerController:
  继续负责真实 preload controller 生命周期。
  继续负责 token / stale task 隔离。
  继续负责 preload hit promote。
  不负责 debounce。
  不负责方向判断。
```

播放意图边界：

```text
preload success 不能修改 wantsToPlay。
preload success 不能调用 play。
只有 playVideo 命中 preloaded controller 时才能 promote。
promote 后是否播放仍由本次 playVideo 的播放意图决定。
```

资源边界：

```text
active controller 最多 1 个。
有效 preload controller 最多 1 个。
pending preload 只是尚未执行的调度，不是 controller。
已启动的旧 preload initialize 迟到后，继续沿用 12.5 过期任务语义处理。
```

Feed 数据边界：

```text
分页 append 不视为换源。
分页 append 不强制取消方向状态。
分页 append 不强制清理当前 preload。
刷新、搜索结果切换、筛选变化或重新加载造成 items 非追加式替换时，视为换源。
换源时必须取消 pending preload，并清理旧 preload。
```

## 执行步骤

### 1. 增加 pending preload 调度状态

文件：

```text
lib/features/feed/coordinators/feed_playback_coordinator.dart
```

新增字段：

```dart
static const Duration _preloadScheduleDelay = Duration(milliseconds: 120);

Timer? _pendingPreloadTimer;
int _preloadScheduleToken = 0;
bool _isDisposed = false;
```

新增方法：

```dart
void _cancelPendingPreloadSchedule() {
  _preloadScheduleToken++;
  _pendingPreloadTimer?.cancel();
  _pendingPreloadTimer = null;
}
```

约束：

```text
每次 current change 入口处都先取消旧 pending preload。
pending preload 取消后不得再调用 preloadVideo。
provider / coordinator dispose 时必须取消 pending preload。
provider / coordinator dispose 后，Timer 回调不能再访问 ref。
```

### 2. 调整 preload 调度入口

将 `_schedulePreload` 保持为单候选调度，但增加稳定窗口。

`_schedulePreload` 建议改为 `void`，并只接收非空 `VideoFeedItem`。它只负责注册或取消调度，不表示 preload 已完成，也不应该被调用方 await 成“完整预加载流程”。

实现约束：

```text
_schedulePreload 只负责“取消旧 pending + 注册新 pending”，不得承担 preload 完成语义。
调用方不得依赖它完成 preload 初始化。
current change 主路径不依赖 _schedulePreload(null)。
无候选清理在 current change 入口完成。
```

推荐语义：

```text
candidate != null:
  取消上一轮 pending preload。
  记录本轮 token。
  启动 120ms timer。
  timer 到期后校验 token。
  timer 到期后清理 _pendingPreloadTimer。
  coordinator 已 dispose 时直接 return。
  token 仍最新才调用 preloadVideo(candidate)。
```

伪代码：

```dart
void _schedulePreload(VideoFeedItem candidate) {
  final playerController = _ref.read(playerControllerProvider.notifier);

  _cancelPendingPreloadSchedule();
  final token = _preloadScheduleToken;

  _pendingPreloadTimer = Timer(_preloadScheduleDelay, () {
    if (_pendingPreloadTimer != null) {
      _pendingPreloadTimer = null;
    }

    if (_isDisposed || token != _preloadScheduleToken) {
      return;
    }

    unawaited(playerController.preloadVideo(candidate));
  });
}
```

注意：

```text
active playVideo / stop 流程不等待这个 timer。
preloadVideo 仍然 fire-and-forget。
disposePreload 也不应阻塞新的 debounce 注册。
disposePreload 内部仍必须保持幂等，并继续由 PlayerController 保证过期任务隔离。
```

### 3. 换源时取消 pending preload

12.7 已在 FeedPlaybackCoordinator 内部通过 Feed item id 前缀判断 items 是否为非追加式替换。

12.8 在检测到换源时补充：

```text
取消 pending preload。
重置方向状态。
fire-and-forget 调用 disposePreload 清理旧 preload。
disposePreload 只用于使旧 preload 业务槽位失效，不阻塞本次 current change 的 active 播放与新 candidate debounce 注册。
如果换源后本次 candidate 为空，不再重复调用 disposePreload。
本次 current change 继续按 unknown 方向重新计算 candidate。
新 candidate 仍走 120ms 稳定窗口。
disposePreload 不应阻塞新 candidate 的 debounce 注册。
```

分页追加时：

```text
不取消 pending preload。
不重置方向。
不强制清理当前 preload。
但如果当前 index 或方向导致 candidate 变化，仍然按正常 current change 流程刷新 preload 调度。
```

### 4. 保持 PlayerController 不变

本阶段原则上不修改：

```text
PlayerController.preloadVideo
PlayerController.disposePreload
PlayerController._canPromotePreload
PlayerController._tryPromotePreloadToActive
PreloadControllerStatus
```

已启动 preload 的过期、失败、迟到完成、dispose-once 仍由 12.5 的 PlayerController 语义保证。

如果实现过程中发现 `disposePreload` 在 idle 或 pending initialize 场景不够幂等，只做必要 bug fix，不扩展多状态机。

### 5. 不扩展正式性能报表

12.8 不要求扩展完整 report。

测试重点验证：

```text
preloadVideo 实际调用次数减少。
最终有效 preload 目标指向最新候选，或在初始化失败时进入该候选对应的 failed 状态。
旧 pending candidate 没有创建 controller。
无候选和换源会立即清理 preload。
```

完整性能收益、命中率、p50 / p90 / p95 对比进入 12.9。

## 阶段结论

12.8 不做多候选和复杂 dispose 策略。

本阶段只做一件小而确定的事：

```text
在单 preload、方向感知、preload hit promote 都不变的前提下，
给 preload 调度增加短稳定窗口，
减少快速滑动时马上过期的后台初始化。
```

完成后，12.9 可以用更干净的数据评估：

```text

```

## 开发记录

实现时发现 `handleFeedCurrentChanged` 内部存在 autoplay grace。如果只在 `_schedulePreload` 内取消旧 pending preload，旧 timer 可能在新的 current change 等待 grace 期间先触发。

因此实现约束补充为：

```text
current change 入口处必须先取消旧 pending preload。
active playVideo / stop 完成后，再为本次最新 candidate 注册新的 pending preload。
```

这样可以保证快速滑动时旧 candidate 不会抢在新 current change 的 active 流程前启动后台 preload。
