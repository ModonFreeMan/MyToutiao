# 12.5 单 preload 过期任务和资源上限

本文档定义第 12.5 阶段的 Feed 单 preload 过期任务隔离、资源上限和快速滑动保护方案。

12.4 已经让 `PlayerController.preloadVideo(item)` 具备真实 preload controller 生命周期：

```text
创建后台 VideoPlayerController
完成 initialize
保留为 preload controller
不 play
不绑定为 active
不修改 PlayerState
```

12.5 在此基础上继续收紧运行时约束，重点回答：

```text
快速滑动时，旧 preload initialize 迟到后会不会覆盖最新 preload 状态？
后台 preload 会不会和 active 初始化争抢关键路径？
PlayerController 对外可见的 controller 资源预算是否始终可控？
离开保留范围的 preload 是否会被释放？
用户暂停意图是否会被 preload 逻辑误恢复？
```

本阶段仍不实现 preload 命中热切换。preloaded controller 升级为 active controller 进入 12.6。

## 目标

```text
确保快速滑动时，旧 preload 完成后不会污染当前预加载状态。
第一版只允许 active + preload 两类 controller 存活。
同一时间对外有效的 preload 业务槽位最多 1 个。
同一时间只有 1 个 preload initialize 任务拥有写回 preload 状态的资格。
过期 initializing 任务可以短暂存在，但不属于有效业务槽位。
当前 active 初始化优先级高于任何后台 preload。
预加载只代表资源准备，不代表用户希望播放。
```

本阶段要把 12.4 的最小异步安全边界升级为清晰、可测试、可回归的资源预算策略。

## 非目标

```text
不实现 preloaded controller -> active controller 的热切换。
不实现 preload hit / miss 的完整性能收益闭环。
不实现方向感知预加载。
不实现 current - 1 / current + 2 多候选队列。
不实现平台层强取消 initialize。
不承诺任何瞬间平台层绝对只有 2 个 VideoPlayerController 实例。
```

这些能力进入后续阶段：

```text
12.6 预加载命中热切换
12.7 方向感知预加载扩展
12.8 快速滑动取消和多候选 dispose 策略
12.9 预加载效果评估与回归验证
```

## 当前基础

12.4 后 `PlayerController` 已具备以下内部字段和状态：

```dart
VideoPlayerController? _preloadController;
String? _preloadVideoId;
VideoQuality? _preloadSelectedQuality;
PreloadControllerStatus _preloadStatus = PreloadControllerStatus.idle;
int _preloadToken = 0;
```

并暴露测试用只读 getter：

```dart
String? get preloadVideoId;
bool get hasPreloadController;
bool get isPreloadInitialized;
PreloadControllerStatus get preloadStatus;
```

12.4 已有的最小保护：

```text
preloadVideo 启动新任务前递增 token。
disposePreload 递增 token，使旧任务过期。
initialize success / failure 回来后校验 token、controller identity 和 videoId。
过期 success 不能把 preloadStatus 改为 preloaded。
过期 failure 不能把当前 preloadStatus 改为 failed。
过期任务只能 dispose-once 自己创建的 controller。
```

12.5 需要在这些基础上补齐更明确的调度语义、资源上限口径和快速滑动测试。

## 状态语义

本阶段继续把 preload 状态限制在 `PlayerController` 内部，不进入 `PlayerState`。

```dart
enum PreloadControllerStatus {
  idle,
  initializing,
  preloaded,
  failed,
}
```

状态约束：

```text
idle:
  没有对外有效的 preload 目标。
  _preloadVideoId == null
  _preloadController == null

initializing:
  当前存在 1 个对外有效的 preload initialize 任务。
  _preloadVideoId != null
  _preloadController != null

preloaded:
  当前存在 1 个已完成 initialize 的 preload controller。
  _preloadVideoId != null
  _preloadController != null
  _preloadController.value.isInitialized == true

failed:
  最近一次对外有效的 preload initialize 失败。
  _preloadVideoId != null
  _preloadController == null
  允许同一 item 下次重试。
```

关键区分：

```text
isInitialized:
  某个 controller 底层是否完成 initialize。

isPreloaded:
  preload 槽位是否有可复用的已初始化资源。

isVisible:
  Feed 当前项是否可见。

wantsToPlay:
  业务播放意图，表示用户或 Feed 自动播放策略希望当前 active 播放。

isPlaying:
  底层 active controller 已确认进入播放状态。
```

约束：

```text
preload 完成只能改变 preload 内部状态。
preload 完成不能修改 wantsToPlay。
preload 完成不能修改 isPlaying。
preload 完成不能修改 active videoId。
preload 完成不能触发 UI 绑定。
```

## 资源预算口径

12.5 的资源预算分为两层。

### 业务槽位预算

对 `PlayerController` 外部和业务调度层可见的槽位必须满足：

```text
active controller 最多 1 个。
preload controller 最多 1 个。
controller 总数默认不超过 2 个。
只有 1 个 preload initialize 任务拥有写回 preload 状态的资格。
```

这里的“有效”表示：

```text
当前 token 匹配。
当前 preloadVideoId 匹配。
当前 _preloadController identity 匹配。
该任务允许写回 preloadStatus。
```

### 平台实例预算

由于 `video_player` 的 initialize 不能可靠同步取消，快速滑动时可能短暂出现：

```text
active controller
最新 preload controller
旧过期 initializing controller
```

12.5 不承诺平台层任意瞬间绝对只有 2 个实例，但必须承诺：

```text
旧过期 initializing controller 不再属于业务 preload 槽位。
旧过期 controller 完成后只能释放自己。
旧过期 controller 不能覆盖当前 preload 状态。
旧过期 controller 不能影响 active 播放状态。
旧过期 controller 最终会被 dispose-once。
旧过期 controller 的存在不违反“只有 1 个任务有写回资格”的业务预算。
```

如果要进一步限制平台层绝对实例数量，需要后续在 12.8 引入取消队列、dispose 队列或更强的资源闸门。

## 核心策略

### active 优先

当前可见视频永远优先于后台 preload。

```text
active 播放 / 初始化链路优先。
preload 调度随后执行。
如果 active 目标等于当前 preload 目标，12.5 仍按 12.4 语义清理 preload 后走原 active 初始化流程。
preload initialize 不得阻塞 active playVideo 的关键路径。
```

推荐在 `FeedPlaybackCoordinator._schedulePreload` 中把真实 preload 改为 fire-and-forget：

```dart
Future<void> _schedulePreload(VideoFeedItem? candidate) async {
  final playerController = _ref.read(playerControllerProvider.notifier);
  if (candidate == null) {
    await playerController.disposePreload();
    return;
  }

  unawaited(playerController.preloadVideo(candidate));
}
```

这样 `handleFeedCurrentChanged` 不会因为后台 preload initialize 变慢而拖住当前 Feed 切换和 active 起播。

如果测试需要等待 preload 结束，测试应直接调用 `PlayerController.preloadVideo`，或提供等待状态进入 `preloaded / failed / idle` 的 helper。

### 单后台任务

`preloadVideo(item)` 的语义：

```text
如果 item 是当前 active video，跳过。
如果 item 已经是当前 preload video 且状态为 initializing / preloaded，跳过。
如果 item 是 failed 目标，允许重试。
如果存在旧 preload 槽位，先使旧任务过期并清空业务槽位。
启动新 preload 前递增 token。
新任务只有 token / controller / videoId 都匹配时才能写回状态。
任何过期任务即使 initialize 成功或失败，也只能进入自己的释放路径。
```

`disposePreload()` 的语义：

```text
递增 token。
清空 preload 业务槽位。
如果旧 controller 已 preloaded，释放旧 controller。
如果旧 controller 仍 initializing，不等待 initialize 完成。
旧任务迟到后由自己的过期分支释放。
disposePreload 对 initializing controller 只负责 invalid token 和清业务槽位。
disposePreload 不直接 dispose 仍在 initializing 的 controller。
```

释放责任必须统一：

```text
每个 preloadVideo 创建的 controller 必须由该任务闭包持有局部引用。
每个 preload 任务必须使用 dispose-once guard。
无论 initialize success / failure / throw / expired，都必须走同一个 dispose-once 方法。
同一个 controller 不能既被 disposePreload 同步 dispose，又被迟到任务再次 dispose。
provider dispose 时也只能通过同一套 dispose-once 路径释放 preload controller。
```

### 快速滑动隔离

快速滑动会触发连续候选变化：

```text
preload A initializing
current index 改变
preload B initializing
A initialize success 迟到
B initialize success
```

必须保证：

```text
A success 不能把 preloadVideoId 改回 A。
A success 不能把 preloadStatus 改成 preloaded。
A failure 不能把 B 的状态改成 failed。
A 只能 dispose 自己创建的 controller。
B 仍然是当前唯一有效 preload 槽位。
```

连续清空也要成立：

```text
preload A initializing
滑到列表尾部，无候选
disposePreload 清空槽位
A initialize success 迟到
```

必须保证：

```text
preloadVideoId 仍为 null。
preloadStatus 仍为 idle。
A controller 被 dispose-once。
不会产生新的 active 播放。
不会修改 PlayerState。
```

### 暂停意图保护

用户主动暂停后：

```text
wantsToPlay == false
```

后台 preload 的任何 success / failure / dispose 都不得：

```text
调用 resume。
调用 play。
修改 wantsToPlay。
通过 ensurePlaybackIntent 间接恢复播放。
```

12.5 不实现 promote，因此 preload 命中时仍不会复用并播放。12.6 实现 promote 时必须继续沿用本阶段的播放意图约束：

```text
promote 后是否 play 由 wantsToPlay 决定。
```

## PlayerController 设计

### preloadVideo

保留 12.4 的 token 校验，并补充更明确的可测试行为：

```text
重复请求同一 initializing / preloaded item 不创建新 controller。
failed item 再次 preload 时会重新创建 controller。
新 item 到来时，旧槽位立刻失效。
旧 initializing 不阻塞新 preload 启动。
旧 preloaded 会被立即 dispose。
新任务完成前，外部只看到新 item 的 preloadVideoId 和 initializing 状态。
```

建议增加内部调试计数，供测试或后续观测使用：

```dart
int _expiredPreloadCompletions = 0;
int _expiredPreloadFailures = 0;
int _preloadDisposeCount = 0;
```

如不希望暴露生产 getter，可以仅在 fake platform 的测试记录中验证 dispose 行为。

### disposePreload

`disposePreload` 必须是幂等的：

```text
idle 状态下重复调用不会抛错。
同一个 controller 不会被 dispose 两次。
pending initialize 迟到后不会再次写回 idle 以外状态。
```

### playVideo

`playVideo(item)` 开头保留防护：

```text
如果 item.id == _preloadVideoId：
  先清理 preload 业务槽位并递增 token。
  再走原 active 初始化流程。
```

12.5 仍然不 promote：

```text
preload 命中不会加速 active 起播。
preload 命中不会复用 controller。
preload 命中不会自动播放。
```

这么做是为了把过期和资源预算先稳定下来，避免 12.6 热切换同时引入过多变量。

## FeedPlaybackCoordinator 设计

候选选择策略继续沿用 12.3：

```text
只从 currentIndex + 1 向后找第 1 个 VideoFeedItem。
非视频 item 不占用 preload 名额。
找不到候选时 disposePreload。
```

12.5 建议调整调度方式：

```text
active 播放 / stop 继续 await。
preloadVideo 改为 unawaited。
disposePreload 仍可以 await，确保列表尾部或无候选时槽位及时清空。
```

原因：

```text
preload initialize 是后台资源准备，不应拖慢当前可见项处理。
disposePreload 清空业务槽位很快，await 更利于尾部状态一致。
旧 initializing controller 的实际 dispose 仍由迟到分支处理，不阻塞 active。
```

如果改为 fire-and-forget，必须满足：

```text
preloadVideo 对外 Future 不允许因为 initialize / dispose / stale completion 抛出未捕获异常。
preloadVideo 内部必须吞掉 preload initialize 失败，并记录 failed 状态。
所有 preload 失败必须被内部捕获，并转成 failed / expired / idle 状态。
FeedPlaybackCoordinator 测试不能假设 handleFeedCurrentChanged 返回时 preload 一定完成。
FeedPlaybackCoordinator 不负责 catch preloadVideo 异常。
```

如果后续不希望 `PlayerController.preloadVideo` 承诺永不抛出 preload 异常，则 `_schedulePreload` 必须显式兜底：

```dart
unawaited(
  playerController.preloadVideo(candidate).catchError(
    (Object error, StackTrace stack) {
      // Log only. Preload errors must not affect active playback.
    },
  ),
);
```

本阶段推荐由 `PlayerController` 内部吞掉 preload 异常，因为 preload 是后台优化能力，不应该污染 active 链路。

## 阶段结论

12.5 的重点不是让 preload 立刻产生起播收益，而是把后台资源准备变成一个预算清晰、过期安全、不会污染播放意图的能力。

本阶段完成后，系统应具备：

```text
FeedPlaybackCoordinator 继续只选择 current + 1 单候选。
PlayerController 始终只暴露 1 个有效 preload 槽位。
快速滑动产生的旧 preload 迟到结果会被安全忽略。
旧 preload controller 最终会被释放且不会重复 dispose。
active 播放链路优先级高于后台 preload。
用户播放意图不受 preload 成败影响。
```

在这个基础上，12.6 才能安全把 `preloaded controller` promote 为 `active controller`，并开始评估真实起播收益。
