# 4 页面层承担业务编排

## 当前问题

Feed 与搜索结果的跨模块播放流程原本分散在页面层：

- `FeedPage` 监听当前内容变化后，直接决定播放视频或停止播放器。
- `SearchResultPage` 点击搜索结果后，直接调用 Feed 定位和 Player 播放。
- `RelatedSearchEntry` 在跳转搜索前后，也直接参与暂停与恢复播放流程。

这些逻辑让 Page 不只负责界面展示，还承担了“当前内容怎么播放、搜索结果怎么定位并播放、页面遮挡后如何暂停恢复”等业务流程编排职责。随着 Feed、Search、Player 的交互继续增加，页面层会越来越难维护，也会让模块边界变得模糊。

## 优化目标

将跨模块播放流程统一收口到业务编排层，让 Page 只负责用户交互和界面状态转发。

优化后应达到：

- 页面层不再直接决定 Feed 与 Player 的跨模块业务流程。
- Feed、Search、Player 之间的协作由专门的 `FeedPlaybackCoordinator` 统一处理。
- 当前 Feed 项变化、搜索结果点击、页面遮挡与恢复等流程具备一致入口。
- 架构文档中的职责说明、Provider、目录结构、核心流程和依赖关系与代码实现保持一致。

## 优化方案

新增 `FeedPlaybackCoordinator`，作为 Feed、Search、Player 之间的业务编排入口。

核心流程调整为：

```text
FeedPage / RelatedSearchEntry / SearchResultPage
  -> FeedPlaybackCoordinator
  -> FeedViewModel
  -> PlayerController
```

`FeedPlaybackCoordinator` 承担以下职责：

- `handleFeedCurrentChanged()`：Feed 当前内容变化后，统一决定播放视频或停止播放器。
- `handleSearchResultSelected()`：搜索结果点击后，统一负责定位 Feed 并触发播放。
- `handleFeedCovered()`：进入搜索或 Feed 被遮挡时，统一暂停当前播放。
- `handleFeedUncovered()`：返回 Feed 或遮挡解除时，统一恢复合适的播放状态。

页面层调整为：

- `FeedPage` 不再直接调用 `PlayerController`，当前 Feed 项变化后只通知 coordinator。
- `SearchResultPage` 不再直接调用 `FeedViewModel` 和 `PlayerController`，点击结果后改为调用 `handleSearchResultSelected()`。
- `RelatedSearchEntry` 跳转搜索前后的暂停与恢复逻辑改走 coordinator。

保留视频卡片内部播放/暂停、清晰度切换等直接播放器交互，因为这些属于播放器控件自身行为，不属于跨模块业务编排。

## 优化验证

从职责边界验证：

- Page 层只负责界面事件转发，不再直接编排 Feed 与 Player 的跨模块流程。
- 跨模块播放相关入口集中在 `FeedPlaybackCoordinator`。
- Feed、Search、Player 的调用链更清晰，后续新增播放编排逻辑时不需要继续堆到页面层。

从核心场景验证：

- Feed 当前内容切换时，能通过 coordinator 播放视频内容或停止非视频内容。
- 点击搜索结果时，能通过 coordinator 定位目标内容并触发播放。
- 进入搜索页时，当前播放能统一暂停。
- 从搜索页返回 Feed 时，播放状态能统一恢复。

从文档一致性验证：

- 总体架构、Provider 划分、目录结构、核心流程和模块依赖关系均已补充业务编排层。
- 文档描述与当前代码中的 `FeedPlaybackCoordinator` 实现保持一致。
