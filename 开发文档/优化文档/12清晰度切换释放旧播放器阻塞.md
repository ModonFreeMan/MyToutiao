# 12 清晰度切换释放旧播放器阻塞

## 当前问题

在补充清晰度切换集成测试时，发现一个 UI 集成场景下的阻塞风险：

```text
Feed 首屏视频正在播放；
点击“切换清晰度”；
选择 1080P；
业务状态进入清晰度切换流程；
但新清晰度播放器没有继续初始化。
```

单元测试中，`PlayerController.switchQuality()` 可以正常通过；但放到真实 Widget 树里，旧 `VideoPlayer` 仍然挂载在页面上时，清晰度切换会卡在释放旧播放器阶段。

这说明原来的清晰度切换能力虽然在控制器层面可用，但在 UI 集成链路里仍有隐患：旧播放器释放不应阻塞新清晰度播放器的创建。

## 根因分析

原始清晰度切换流程是：

```dart
await _disposeCurrent();

state = state.copyWith(
  selectedQuality: quality,
  isInitializing: true,
  isInitialized: false,
);

final nextController = VideoPlayerController.networkUrl(...);
await nextController.initialize();
```

这里存在两个问题：

```text
1. 先 await 旧播放器 dispose，再更新状态。

   UI 仍然认为旧播放器处于 initialized 状态，
   VideoPlayerView 还可能继续持有旧 controller。

2. 新播放器初始化被旧播放器释放阻塞。

   如果旧 Texture、事件订阅或测试 fake platform 的 stream close
   没有立即完成，switchQuality() 就不会继续创建新 controller。
```

在 Widget 测试环境中，这个问题更容易暴露。`VideoPlayer` widget 仍订阅旧 controller 事件流时，旧平台 stream 的关闭可能等待订阅释放；而订阅释放又依赖 Widget 树先 rebuild 脱离旧 controller。

因此根因可以概括为：

```text
清晰度切换把“释放旧播放器”放在新状态更新和新播放器创建之前，
导致 UI 还没脱离旧 controller 时就等待旧资源释放，
形成切换链路阻塞。
```

## 优化方案

清晰度切换改为先更新业务状态，再非阻塞释放旧播放器。

新的顺序是：

```text
1. 记录当前播放位置和播放意图。
2. 将 PlayerState 更新为目标清晰度 + 初始化中。
3. 让 UI 立即脱离旧 initialized controller。
4. 异步释放旧 controller，不阻塞后续流程。
5. 创建并初始化新清晰度 controller。
6. 恢复原播放位置。
7. 如果切换前仍有播放意图，则继续播放。
```

核心调整：

```dart
state = state.copyWith(
  videoId: item.id,
  selectedQuality: quality,
  isInitializing: true,
  isInitialized: false,
  isPlaying: false,
  wantsToPlay: shouldContinuePlaying,
  currentPosition: previousPosition,
  duration: item.duration,
  clearError: true,
);

await _disposeCurrent(waitForDispose: false);
```

`_disposeCurrent()` 增加参数：

```dart
Future<void> _disposeCurrent({bool waitForDispose = true}) async {
  ...
  final dispose = controller.dispose();
  if (waitForDispose) {
    await dispose;
  } else {
    unawaited(dispose);
  }
}
```

这样清晰度切换不会被旧播放器释放卡住；普通停止、销毁等路径仍保持等待释放完成。

## 修改范围

本次优化修改播放器控制层：

```text
lib/features/player/controllers/player_controller.dart
```

具体调整：

```text
switchQuality()：
  先写入目标清晰度和初始化中状态；
  再非阻塞释放旧 controller；
  继续初始化新 controller。

  新增 waitForDispose 参数；
  默认保持等待释放；
  清晰度切换路径使用非阻塞释放。
```

## 优化后的行为

优化后，清晰度切换行为如下：

```text
1. 用户选择 1080P。
2. PlayerState 立即进入 selectedQuality=1080P、isInitializing=true。
3. UI 不再继续以 initialized 状态持有旧播放器。
4. 旧 controller 异步释放。
5. 新 controller 立即开始初始化。
6. 初始化完成后保留切换前播放进度。
7. 如果切换前用户希望继续播放，则切换后继续播放。
```

最终效果：

```text
清晰度切换不会因为旧播放器释放慢而卡住；
切换后能进入目标清晰度；
播放进度和播放意图保持不变；
普通停止和页面销毁路径仍等待资源释放完成。
```
