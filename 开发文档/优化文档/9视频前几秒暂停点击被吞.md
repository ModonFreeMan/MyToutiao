# 9 视频前几秒暂停点击被吞

## 当前问题

当前播放链路中存在一个交互竞态问题：

```text
用户滑到一个新视频后，
视频刚开始进入播放流程，
此时用户快速点击屏幕中心想要暂停，
但暂停没有稳定生效，视频仍然继续播放。
```

这个问题最初看起来像是播放器状态判断问题：

```text
视频已经初始化并正在播放，
但仍处于 buffering 状态，
点击暂停被误判为 loading 阶段点击并被吞掉。
```

典型状态组合是：

```dart
isInitialized == true
isPlaying == true
isBuffering == true
```

这个状态在真实播放中很常见。视频开始播放后，前几秒或网络波动时仍可能处于 buffering，但用户已经看到视频处于播放态，此时点击当前视频区域应该被理解为暂停意图。

继续验证后发现，问题不止这一层，还包括两个更早的竞态窗口。

第一个窗口发生在初始化尚未完成时：

```dart
isInitializing == true
isInitialized == false
isPlaying == false
```

此时 `playVideo()` 已经把业务当前视频切换为新视频，但底层 `VideoPlayerController.initialize()` 尚未完成。用户再次点击当前视频区域时，直觉上是在取消这次即将开始的自动播放；旧逻辑会把点击当成 loading 阶段点击吞掉，等初始化完成后视频仍然继续自动播放。

第二个窗口发生在快速翻页后：

```text
用户刚滑动到新视频，
PageView 还在 settle，
播放器切换编排也可能还在路上，
此时用户快速点击屏幕中心想暂停当前新视频。
```

这个场景里，点击可能没有进入 `VideoFeedCard` 的内部点击分支。也就是说，问题不只是播放器状态判断，还包括快速翻页过程中点击命中层级不稳定：

```text
Feed 当前项已经变成新视频，
但子卡片 GestureDetector 可能还没有稳定拿到 tap；
随后 Feed 当前项变化触发的自动播放继续执行，
新视频最终仍然开始播放。
```

因此，这个问题可以归纳为三个竞态窗口：

```text
1. 已播放但仍在 buffering：点击暂停被 loading guard 拦截。
2. 已切到当前视频但初始化未完成：点击无法取消 pending autoplay。
3. 快速翻页后 PageView settle：子卡片 tap 可能丢失，自动播放继续执行。
```

## 根因分析

根因一：`FeedPlaybackCoordinator.handleVideoCardTapped` 中的 loading 判定过宽。

旧逻辑中，`isLoadingVideo` 会把以下状态都视为 loading：

```text
当前视频正在初始化
或
当前视频已经初始化但正在 buffering
```

随后通过提前返回吞掉点击：

```dart
if (isLoadingVideo) {
  return;
}
```

这个判断没有区分两类不同场景：

```text
A. 视频还没准备好，不能处理普通播放控制点击。
B. 视频已经在播放，只是仍处于 buffering，可以处理暂停点击。
```

因此，当视频处于：

```dart
isInitialized == true
isPlaying == true
isBuffering == true
```

时，点击当前视频本应触发暂停，却被 `isLoadingVideo` 提前拦截。

根因二：初始化未完成时，底层 controller 尚未挂载，普通暂停无法生效。

在 `isInitializing == true` 且 `isInitialized == false` 的窗口内，播放器底层 controller 还不可用。如果此时只是简单忽略点击，后续 initialize 完成后仍会继续执行自动播放。

因此，这个阶段的用户点击不应该被视为普通 pause，而应该被解释为：

```text
取消这次 pending autoplay。
```

根因三：快速翻页时，点击入口放在卡片内部不够稳定。

快速从图片页滑到新视频页后，`PageView` 可能仍处于 settle 阶段。此时用户点击屏幕中心，点击不一定能稳定进入 `VideoFeedCard` 内部的 `GestureDetector`。

但与此同时，Feed 当前项变化已经触发自动播放。如果没有页面级点击识别和自动播放压制机制，就会出现：

```text
用户已经点了屏幕中心想暂停，
但点击没有进入卡片；
自动播放任务继续完成；
新视频最终仍然播放。
```

## 修复目标

本次修复的目标是：

```text
让用户在视频刚进入播放流程时的暂停/取消意图稳定生效。
```

修复后应满足：

* 当前视频已经初始化并正在播放时，即使仍处于 buffering，也可以点击暂停。
* 当前视频正在初始化但尚未播放时，再次点击可以取消这次 pending autoplay。
* 快速翻页到新视频后，立刻点击屏幕中心可以取消/压制该视频的自动播放。
* 当前视频暂停后，再次点击可以恢复播放。
* 点击非当前视频时，仍按原逻辑切换并播放新视频。
* 当前视频初始化中重复点击不会反复创建播放器。
* loading guard 只保护真正不可直接操作的准备阶段，不吞掉明确的用户暂停/取消意图。

## 修复方案

### 1. 调整当前视频点击的判断优先级

在 `FeedPlaybackCoordinator.handleVideoCardTapped` 中，优先处理当前视频已经播放的情况：

```dart
if (isCurrentVideo && playerState.isInitialized && playerState.isPlaying) {
  await playerController.togglePlayPause();
  return;
}
```

这条逻辑必须发生在 loading guard 之前。

它表达的业务语义是：

```text
只要当前视频已经处于播放态，
用户点击当前视频区域就是暂停意图，
不应该再被 buffering/loading 状态拦截。
```

因此，`isBuffering` 可以继续用于 UI loading 展示，但不能一概作为“禁止处理点击”的依据。

### 2. 初始化未完成时，点击取消 pending autoplay

对于当前视频已经进入初始化流程、但尚未完成初始化的情况，用户再次点击当前视频时，不再简单 return，而是取消这次待完成的自动播放。

该逻辑要保证：

```text
当前视频正在初始化时，
重复点击不会反复创建 VideoPlayerController；
同时，用户可以通过再次点击取消即将开始的自动播放。
```

也就是说，这个阶段的点击语义不是 pause，而是 cancel pending autoplay。

### 3. 在 PageView 外层识别中心视频区域点击

快速翻页场景下，不能只依赖 `VideoFeedCard` 内部的点击识别。

因此，在 `FeedPage` 的 `PageView` 外层增加页面级点击识别：

```text
FeedPage
  -> PageView 外层 Listener 捕获 pointer down/up
  -> 计算 pointer 位移
  -> 位移小于阈值，认为是点击
  -> 判断点击是否落在中心视频区域
  -> 读取当前 Feed item
  -> 如果当前 item 是 VideoFeedItem，调用 FeedPlaybackCoordinator.handleVideoCardTapped
```

这样可以避免快速翻页 settle 期间，子卡片 GestureDetector 没有稳定收到 tap 的问题。

该逻辑的边界是：

```text
明显上下滑动不会被误判为点击；
顶部搜索区和底部信息/进度区域不会触发中心视频点击；
真正的视频中心区域点击不依赖子卡片是否已经稳定命中。
```

### 4. 移除卡片内部视频区域 tap 入口

由于视频区域点击已经上移到 `FeedPage` 页面层统一处理，`VideoFeedCard` 内部不再保留重复的 tap 入口。

这样可以避免：

```text
PageView 外层点击识别
和
VideoFeedCard 内部点击识别
```

重复触发同一次播放控制逻辑。

### 5. 为翻页自动播放增加短保护窗口

对于快速翻页到新视频后的自动播放，`FeedPlaybackCoordinator` 增加短保护窗口。

其语义是：

```text
handleFeedCurrentChanged(video)
  -> 记录本次翻页自动播放候选视频
  -> 短暂延迟后再执行自动播放
  -> 如果保护窗口内用户点击该视频
  -> 取消/压制本次自动播放
```

这个保护窗口只用于处理“刚翻页到新视频后马上点击”的用户意图，不应成为长期全局冷却时间。

同时，该保护窗口不应用于首屏初始视频，避免首屏普通点击被误判为取消自动播放。

## 修改范围

本次修复聚焦在 Feed 播放编排和点击入口层，不修改播放器底层实现。

涉及代码：

* `feed_page.dart`

  * 在 `PageView` 外层增加中心视频区域点击识别。
  * 使用移动阈值过滤上下滑，避免把 swipe 误判为 tap。
  * 当前 item 是视频时，将点击交给 `FeedPlaybackCoordinator`。

* `feed_playback_coordinator.dart`

  * 调整当前视频点击的判断优先级。
  * 已播放状态优先响应暂停。
  * 初始化未完成状态允许取消 pending autoplay。
  * 翻页自动播放增加短保护窗口，允许快速点击压制自动播放。

* `video_feed_card.dart`

  * 移除卡片内部视频区域 tap 入口。
  * 避免与 `FeedPage` 外层点击识别重复触发。

* `feed_playback_coordinator_test.dart`

  * 覆盖 buffering 中的播放态点击可以暂停。
  * 覆盖初始化未完成时点击可以取消 pending autoplay。

* `basic_playback_flow_test.dart`

  * 覆盖快速从图片页滑到新视频后立刻点击，最终新视频不能继续保持播放态。

未扩大修改范围：

* 不修改 `VideoPlayerView` 的渲染逻辑。
* 不修改 `PlayerController.pause()` / `togglePlayPause()` 的底层调用逻辑。
* 不重构 fake video platform 的通用行为。
* 不调整视频卡片视觉展示，除移除重复点击入口外，UI 展示语义保持不变。

## 修复后的行为

修复后，点击行为按照以下优先级处理：

```text
1. 如果点击的不是当前视频，按原逻辑切换并播放该视频。
2. 如果点击的是当前视频，并且已经初始化且正在播放，优先暂停。
3. 如果点击的是当前视频，但初始化尚未完成，取消本次 pending autoplay。
4. 如果仍是其它真正不可操作的 loading 状态，忽略点击。
5. 如果点击的是当前视频，已经初始化但未播放，则恢复播放。
```

快速翻页场景下：

```text
1. 用户从图片页快速滑到新视频页。
2. Feed 当前项变化触发新视频自动播放候选任务。
3. PageView 仍在 settle 时，用户点击屏幕中心。
4. FeedPage 外层 Listener 捕获该中心区域点击。
5. 点击进入 FeedPlaybackCoordinator。
6. Coordinator 取消/压制本次翻页自动播放。
7. 最终新视频不再继续保持播放态。
```

本次修复完成后，播放链路满足以下标准：

```text

当前视频一旦已经初始化并处于播放态，
用户点击暂停必须被响应，

```

## 后续补充：暂停后进度回到 0s

继续手动验证后，又发现一个独立问题：

```text
视频前几秒点击暂停后，
暂停动作已经发生，
但进度会直接回到 0s。
```

这个问题不再属于 Feed 点击编排范围。此时点击意图已经能够送达播放器控制层，新的失败点是：

```text
暂停送达后，PlayerController 同步到的 currentPosition 是否准确。
```

因此本次最终修复边界应明确为：

```text
只修 PlayerController 的暂停进度同步问题；
不再修改 FeedPage / FeedPlaybackCoordinator / VideoFeedCard。
```

红灯测试：

- [player_controller_test.dart](d:/Flutter/VideoPlayer/test/unit/player/player_controller_test.dart:160)

测试模拟：

```text
平台真实播放位置已经到 3s，
但 VideoPlayerController.value.position 缓存仍是 0s，
此时点击暂停。
```

修复前失败结果为：

```text
Expected: Duration:<0:00:03.000000>
Actual:   Duration:<0:00:00.000000>
```

根因是：

```text
PlayerController 在暂停后立刻通过 _syncFromController()
从 VideoPlayerController.value.position 同步进度。

但 value.position 是 video_player 插件侧缓存值，
依赖 position timer 周期刷新。
当前几秒内 timer 尚未刷新时，
缓存 position 仍可能是 0s，
而平台真实播放位置已经前进到 3s。
```

修复方案：

```text
PlayerController.pause()
PlayerController.togglePlayPause() 的暂停分支
```

在执行暂停前，先主动读取：

```dart
await controller.position
```

该 getter 会调用底层 `VideoPlayerPlatform.getPosition()`，比 `controller.value.position` 更接近真实播放位置。

最终同步顺序为：

```text
controller.position
  -> 如果为 null，回退 controller.value.position
  -> 经过 _clampPosition(position, duration)
  -> 写入 PlayerState.currentPosition
```

修复后，暂停动作会保留平台真实播放位置，不会因为插件缓存尚未刷新而把业务进度回写为 0s。

## 后续补充：已播放视频被误判为取消 autoplay

继续验证“滑动到新页面后快速点击暂停”时，又发现一个语义优先级问题：

```text
video_002 已经自动播放成功，
用户快速点击，本应是暂停；
但代码仍把它当成“翻页自动播放保护窗口内的取消 autoplay”，
于是清空了播放器业务状态。
```

红灯测试：

- [basic_playback_flow_test.dart](d:/Flutter/VideoPlayer/test/integration/basic_playback_flow_test.dart:137)

失败现象：

```text
Expected: 'video_002'
Actual: <null>
```

这说明快速点击后没有走“暂停当前视频”的语义，而是走了 `stopIfCurrent(item.id)`，导致 `videoId` 被清空。UI 随后不再认为当前卡片是正在播放的视频，于是表现为回到 0s 并显示封面/初始态。

本次修复不修改：

```text
FeedPage
VideoFeedCard
PageView 外层 Listener
```

点击已经能够送达 `FeedPlaybackCoordinator`，问题发生在点击语义解释阶段。

最终规则：

```text
_isSettlingAutoplay 只能影响“尚未播放成功”的当前视频；
不能影响 already playing 的当前视频。
```

判断优先级固定为：

```text
1. 当前视频已 initialized && playing：
   点击 = pause，保留 videoId 和 currentPosition。

2. 当前视频 isInitializing：
   点击 = cancel pending autoplay，允许 stopIfCurrent。

3. 当前视频未 playing 且仍在 settling autoplay：
   点击 = cancel pending autoplay，允许 stopIfCurrent。

4. 当前视频 initialized && paused：
   点击 = resume。

5. 非当前视频：
   点击 = 切换播放。
```

也就是说：

```text
未播放成功前快速点击：取消 pending autoplay。
已播放成功后快速点击：暂停当前视频，并保留当前进度。
```

最终集成测试验证两层协作：

```text
FeedPlaybackCoordinator：
  不再把已播放视频清空成 null；
  点击后 videoId 仍然是 video_002，isPlaying=false。

PlayerController：
  暂停时读取真实 position；
  currentPosition 保持为 3s，而不是回到 0s。
```

最终目标是：

```text
用户看到新视频即将播放或正在播放时，
点击屏幕中心的暂停/取消意图必须稳定生效，
不能被 buffering、initializing、PageView settle 或自动播放异步任务吞掉。
```
