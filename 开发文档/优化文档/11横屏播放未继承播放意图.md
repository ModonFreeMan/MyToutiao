# 11 横屏播放未继承播放意图

## 当前问题

从 Feed 视频进入横屏播放时，偶现横屏页没有继续自动播放。

典型表现是：

```text
Feed 中视频正在播放；
点击“横屏播放”；
进入横屏页后，画面停在暂停态；
页面中央出现播放按钮，需要用户再次点击才继续播放。
```

这个问题不是普通的播放初始化失败。进入横屏前，`PlayerState.isPlaying` 通常仍是 `true`；问题发生在方向切换、路由挂载、视频渲染面迁移期间。

继续补充横竖屏来回切换测试后，又暴露出两个同源问题：

```text
1. 竖屏处于用户主动暂停状态时进入横屏，
   横屏页会自动取消暂停并开始播放。

2. 横屏处于播放状态时返回竖屏，
   竖屏原本应该继续播放的视频会停在暂停态。
```

这说明问题不只发生在“进入横屏”的单向链路，而是整个横竖屏渲染切换过程中，播放事实状态和播放意图状态没有被完整继承。

## 根因分析

更深层原因是当前状态机原本没有区分：

```text
播放事实状态：底层播放器此刻是否真的在播放。
播放意图状态：业务或用户是否希望当前视频继续播放。
```

原先只有 `isPlaying` 一个字段，它同时承担了这两种含义。

横屏入口流程是：

```text
VideoFeedCard._openLandscapePlayer()
  -> FeedPlaybackCoordinator.handleLandscapeRequested(item)
  -> SystemChrome.setPreferredOrientations(...)
  -> Navigator.pushNamed(...)
  -> LandscapePlayerPage.initState()
```

`handleLandscapeRequested()` 会在进入横屏前检查一次播放状态：

```dart
if (!isCurrentVideo || !playerState.isInitialized) {
  await playerController.playVideo(item, forceRestart: true);
} else if (!playerState.isPlaying) {
  await playerController.resume();
}

playerController.setLandscapeRendering(true);
```

问题在于，这次 `resume()` 或状态检查发生得太早。

真实设备上，方向切换或 Texture 渲染面迁移期间，底层播放器可能发出一次 `isPlaying=false`。`PlayerController._syncFromController()` 会把这个 native 回调同步进 `PlayerState.isPlaying`。

由于当时没有单独记录“业务仍希望继续播放”，这个临时 paused 事实状态会覆盖整个播放语义。横屏页挂载后只会根据 `isPlaying=false` 显示播放按钮，不会知道这是渲染迁移造成的临时事实暂停，也不会主动恢复。

因此根因是：

```text
native 回调描述的是事实状态；
横屏切换需要保留的是播放意图；
原状态机把两者混在一个 isPlaying 字段中；
导致切换期间的临时 paused 回调吞掉了“继续播放”的业务意图。
```

在引入 `wantsToPlay` 后，第一版修复仍有两个遗漏：

```text
1. handleLandscapeRequested() 仍然用 !isPlaying 判断是否 resume。

   用户主动暂停时：
     isPlaying=false
     wantsToPlay=false

   但旧逻辑只看到 !isPlaying，
   于是调用 resume()，
   把用户暂停意图错误改成了播放意图。

2. 播放意图只在 LandscapePlayerPage 中被消费。

   横屏返回竖屏时，如果 native 在 pop 或方向恢复过程中上报 isPlaying=false，
   状态会变成：
     isPlaying=false
     wantsToPlay=true
     isLandscapeRendering=false

   此时横屏页已经退出，竖屏 VideoPlayerView 原本没有消费播放意图，
   所以竖屏视频会停在暂停态。
```

## 修复方案

将播放状态拆成两个维度：

```text
isPlaying:
  播放事实状态。
  只表示底层播放器当前是否正在播放。

wantsToPlay:
  播放意图状态。
  表示业务或用户是否希望当前视频继续播放。
```

新增字段：

```dart
final bool wantsToPlay;
```

状态规则：

```text
playVideo() / resume():
  wantsToPlay = true

pause() / 用户主动暂停:
  wantsToPlay = false

native isPlaying=false 回调:
  只更新 isPlaying
  不清除 wantsToPlay

stop() / stopIfCurrent() / 初始化失败:
  清空或重置播放意图
```

新增恢复播放意图的方法：

```dart
Future<void> ensurePlaybackIntent(String videoId) async {
  if (state.videoId != videoId ||
      !state.isInitialized ||
      state.isPlaying ||
      !state.wantsToPlay) {
    return;
  }

  await resume();
}
```

横屏页监听 `playerControllerProvider`。当满足以下条件时，下一帧恢复播放：

```text
当前 videoId 匹配；
处于横屏渲染态；
播放器已初始化；
事实状态 isPlaying=false；
播放意图 wantsToPlay=true。
```

也就是：

```text
如果事实暂停了，但业务仍希望播放，
横屏页挂载后消费这个播放意图，
调用 ensurePlaybackIntent(videoId) 恢复播放。
```

横屏入口改为不再基于 `!isPlaying` 无条件恢复：

```dart
if (!isCurrentVideo || !playerState.isInitialized) {
  await playerController.playVideo(item, forceRestart: true);
} else {
  await playerController.ensurePlaybackIntent(item.id);
}

playerController.setLandscapeRendering(true);
```

这样进入横屏时只继承播放意图：

```text
wantsToPlay=true:
  如果事实暂停，则恢复播放。

wantsToPlay=false:
  保持用户主动暂停，不自动播放。
```

横屏关闭改为异步，并在关闭横屏渲染状态后消费一次播放意图：

```dart
Future<void> handleLandscapeClosed() async {
  final playerState = _ref.read(playerControllerProvider);
  final playerController = _ref.read(playerControllerProvider.notifier);

  playerController.setLandscapeRendering(false);

  final videoId = playerState.videoId;
  if (videoId != null) {
    await playerController.ensurePlaybackIntent(videoId);
  }
}
```

同时，竖屏 `VideoPlayerView` 也监听 `playerControllerProvider`。当满足以下条件时，下一帧恢复播放：

```text
当前 videoId 匹配；
不处于横屏渲染态；
播放器已初始化；
事实状态 isPlaying=false；
播放意图 wantsToPlay=true。
```

也就是：

```text
横屏页负责横屏阶段消费播放意图；
竖屏 VideoPlayerView 负责竖屏阶段消费播放意图。
```

这样可以覆盖返回竖屏后，native 在路由 pop 或方向恢复尾声才补发 paused 事件的情况。

## 修复后的行为

修复后，横屏进入流程中的播放语义变为：

```text
1. Feed 视频正在播放时，PlayerState.isPlaying=true，wantsToPlay=true。
2. 点击横屏后，coordinator 标记进入横屏渲染态。
3. 如果方向切换或渲染面迁移导致 native 上报 paused，
   PlayerState.isPlaying 会变成 false，
   但 wantsToPlay 仍保持 true。
4. 横屏页挂载并监听到：
   当前视频已初始化 + 横屏渲染中 + 事实暂停 + 意图播放。
5. 横屏页调用 ensurePlaybackIntent(videoId)。
6. PlayerController 通过 resume() 恢复播放。
```

竖屏暂停进入横屏的播放语义变为：

```text
1. 用户在竖屏主动暂停。
2. PlayerState.isPlaying=false，wantsToPlay=false。
3. 点击横屏后，coordinator 调用 ensurePlaybackIntent(videoId)。
4. ensurePlaybackIntent 发现 wantsToPlay=false，直接返回。
5. 横屏页保持暂停态。
```

横屏播放返回竖屏的播放语义变为：

```text
1. 横屏中视频正在播放。
2. PlayerState.isPlaying=true，wantsToPlay=true。
3. 点击返回竖屏。
4. coordinator 关闭 isLandscapeRendering。
5. 如果返回过程中 native 上报 paused，
   PlayerState.isPlaying=false，但 wantsToPlay 仍为 true。
6. 竖屏 VideoPlayerView 监听到：
   当前视频已初始化 + 竖屏渲染中 + 事实暂停 + 意图播放。
7. 竖屏 VideoPlayerView 调用 ensurePlaybackIntent(videoId)。
8. PlayerController 通过 resume() 恢复播放。
```

最终效果：

```text
进入横屏播放时，
即使底层播放器在切换过程中短暂上报 paused，
也会继承“继续播放”的业务意图，
横屏页最终自动恢复播放。

竖屏暂停进入横屏时，
会继承“保持暂停”的用户意图，
横屏页不会自动播放。

横屏播放返回竖屏时，
即使底层播放器在返回过程中短暂上报 paused，
竖屏视图也会继承“继续播放”的业务意图，
最终恢复播放。
```

## 通过标准

本问题修复完成后，需要满足：

```text
isPlaying 只表达底层事实播放状态。
wantsToPlay 表达业务播放意图。
native paused 回调不能清除播放意图。
用户主动暂停必须清除播放意图。
横屏页能在事实暂停但意图播放时恢复播放。
竖屏页能在事实暂停但意图播放时恢复播放。
竖屏暂停进入横屏时不能自动播放。
横屏播放返回竖屏时不能停在暂停态。
退出横屏后横屏渲染状态正常关闭。
不破坏 Feed 遮挡恢复、质量切换、点击暂停/播放等原有行为。
```
