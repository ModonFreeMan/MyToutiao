# 10 暂停后恢复播放进度条短暂回退

## 当前问题

视频点击暂停后，再次点击恢复播放时，底部进度条会短暂回退，并且播放也会从回退后的位置继续。

典型现象是：

```text
暂停时进度停在 8.400s，
再次点击播放后，
进度条回退到 8.100s 附近，
播放从 8.100s 附近继续。
```

这个问题不是只发生在视频前几秒，其它有效播放位置也会出现。多次重复暂停/播放时，回退幅度基本稳定，说明它更像是播放器缓存位置与平台真实位置之间的滞后，而不是用户拖动进度条导致的 seek 精度问题。

## 根因分析

问题发生在暂停和恢复播放的 position 来源不一致。

暂停分支会主动读取平台真实位置：

```dart
final position = await controller.position ?? controller.value.position;
await controller.pause();
_syncFromController(currentPosition: position);
```

这里的 `controller.position` 会读取底层平台位置，因此暂停时 `PlayerState.currentPosition` 能保存到较新的真实位置。

但恢复播放分支原本会在 `play()` 后直接同步 `controller.value.position`：

```dart
await controller.play();
_syncFromController();
```

`controller.value.position` 是 `video_player` 插件维护的缓存值，可能比平台真实位置落后一小段。因此恢复播放后，较新的 `PlayerState.currentPosition` 会被旧缓存覆盖，进度条就会短暂回退。

继续验证后发现，问题不止是 UI 状态回退。即使临时保住 `PlayerState.currentPosition`，底层播放器位置仍可能停留在旧缓存位置附近。如果恢复播放时没有把底层播放器重新校准到暂停时的真实位置，实际播放也会从旧位置继续。

因此根因分为两层：

```text
1. 业务状态回退：
   恢复播放后使用落后的 controller.value.position 覆盖了 PlayerState.currentPosition。

2. 底层播放位置未校准：
   暂停时只把真实位置保存到业务状态；
   恢复播放时没有 seek 到该位置；
   导致实际播放仍可能从旧缓存位置继续。
```

另外，真实点击路径不是 `resume()`，而是：

```text
FeedPlaybackCoordinator.handleVideoCardTapped()
  -> playerController.togglePlayPause()
  -> togglePlayPause() 的恢复播放分支
```

所以只修 `resume()` 不足以覆盖手动点击暂停后恢复播放的场景。`resume()` 仍需要覆盖，但它主要对应 Feed 遮挡恢复、横屏恢复等显式恢复场景。

## 修复方案

修复思路是统一恢复播放入口，并在恢复播放前校准底层播放器位置。

新增统一恢复方法：

```dart
Future<void> _playPreservingProgress(VideoPlayerController controller) async {
  final currentPosition = state.currentPosition;
  final cachedPosition = controller.value.position;

  if (cachedPosition < currentPosition) {
    await controller.seekTo(currentPosition);
  }

  await controller.play();

  final nextCachedPosition = controller.value.position;
  _syncFromController(
    currentPosition: nextCachedPosition < currentPosition
        ? currentPosition
        : nextCachedPosition,
  );
}
```

它表达的规则是：

```text
恢复播放前，先记录业务状态中的 currentPosition。
如果 controller.value.position 落后于 currentPosition，
说明插件缓存位置较旧，需要先 seekTo(currentPosition) 校准底层播放器。

play() 后再次同步时，
如果 cached position 仍落后，就继续保留 currentPosition；
如果 cached position 已经更新到更靠后的位置，则允许向前同步。
```

`togglePlayPause()` 的恢复播放分支改为：

```dart
await _playPreservingProgress(controller);
```

`resume()` 也改为：

```dart
await _playPreservingProgress(controller);
```

暂停分支保持不变，继续使用平台真实位置：

```dart
final position = await controller.position ?? controller.value.position;
await controller.pause();
_syncFromController(currentPosition: position);
```

## 验证覆盖

新增测试覆盖：

```text
resume keeps paused progress before cached position updates
resume keeps non-initial paused progress before cached position updates
tap resume does not move progress backward by stale cached offsets
tap resume realigns playback to paused position when cache lagged
```

测试覆盖的关键场景：

```text
3.200s 不应被 3.000s 覆盖。
8.400s 不应被 8.100s 覆盖。
点击暂停后再次点击播放，不应从 3.140s 回退到 2.870s。
点击暂停后再次点击播放，不应从 6.895s 回退到 6.730s。
点击暂停后再次点击播放，不应从 10.810s 回退到 10.420s。
当 cached position 落后于暂停 position 时，恢复播放必须 seek 到暂停 position。
```


## 修复后的行为

修复后，暂停与恢复播放的行为如下：

```text
1. 暂停时，优先读取平台真实 position，并保存到 PlayerState.currentPosition。
2. 恢复播放时，如果 controller.value.position 落后于 PlayerState.currentPosition，
   先 seekTo(PlayerState.currentPosition) 校准底层播放器。
3. 再执行 play()。
4. play() 后同步状态时，不允许旧 cached position 覆盖较新的 currentPosition。
5. 如果 cached position 已经更靠后，则允许正常向前同步。
```

最终效果：

```text
点击暂停后再次点击播放，
进度条不会往前跳，
播放也会从暂停时记录的真实位置继续。
```

## 通过标准

本问题修复完成后，需要满足：

```text
业务进度不被落后的 controller.value.position 覆盖。
底层播放器恢复播放前会在必要时校准到暂停 position。
普通点击暂停/播放路径 togglePlayPause() 正常。
显式恢复播放路径 resume() 正常。
不引入额外 UI、Feed 编排或进度条逻辑修改。
```
