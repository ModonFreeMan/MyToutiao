# 12.4 PlayerController preload 能力

本文档定义第 12.4 阶段的 PlayerController 真实预加载能力。

12.3 已完成 current + 1 preload candidate 选择，并通过 `PlayerController.preloadVideo(item)` 记录候选 `videoId`。

12.4 在同一入口上继续向下实现真实 preload controller 生命周期：

```text
创建后台 VideoPlayerController
完成 initialize
保留为 preload controller
不 play
不绑定为 active
不触发 UI 切换
```

本阶段只解决“资源提前初始化”问题，不实现命中后的热切换。preload 命中 promote 进入 12.6。

注意：

```text
12.4 虽然会真实 initialize preload controller，
但命中 preload 时仍然 dispose preload 并走原 active 初始化流程。
因此本阶段不宣称起播性能收益。
12.4 的收益是建立可隔离、可释放、可测试的 preload controller 生命周期。
真正的性能收益需要 12.6 热切换后才能评估。
```

## 目标

```text
给 PlayerController 增加独立的 preload controller 管理能力。
preloadVideo(item) 不再只是记录 videoId，而是创建并 initialize preload controller。
active controller 和 preload controller 必须相互独立。
preload initialize 成功后只表示资源准备完成。
preload initialize 失败不能影响当前 active 播放状态。
disposePreload() 可以释放当前 preload controller。
```

## 非目标

```text
不实现 preloaded controller -> active controller 的热切换。
不改变 FeedPlaybackCoordinator 的 current + 1 候选选择策略。
不实现方向感知、多候选队列或 current - 1 / current + 2。
不实现复杂 generation 队列、取消队列或多候选过期策略。
不实现 preload hit / miss 完整指标闭环。
不改变 VideoPlayerView 的渲染绑定逻辑。
```

这些能力进入后续阶段：

```text
12.5 单 preload 过期任务和资源上限
12.6 预加载命中热切换
12.7 方向感知预加载扩展
```

## 当前基础

12.3 后 `PlayerController` 已有轻量入口：

```dart
String? _preloadVideoId;

String? get preloadVideoId => _preloadVideoId;

Future<void> preloadVideo(VideoFeedItem item) async {
  if (state.videoId == item.id) {
    return;
  }

  _preloadVideoId = item.id;
}

Future<void> disposePreload() async {
  _preloadVideoId = null;
}
```

12.4 需要在此基础上扩展真实资源：

```text
_preloadController
_preloadVideoId
_preloadSelectedQuality
_preloadStatus
_preloadToken
```

其中 `_preloadVideoId` 的语义从“候选 id”升级为：

```text
当前 preload 槽位正在处理或最近失败的视频 id。
```

它仍然不代表 active video，也不代表用户播放意图。

## 状态语义

建议在 `PlayerController` 内部定义 preload 状态，而不是放入 `PlayerState`：

```dart
enum PreloadControllerStatus {
  idle,
  initializing,
  preloaded,
  failed,
}
```

内部状态含义：

```text
idle:
  没有 preload 目标，也没有 preload controller。

initializing:
  已创建 preload controller，正在 initialize。

preloaded:
  preload controller 已 initialize 成功，可在后续 12.6 被 promote。

failed:
  最近一次 preload initialize 失败。
  失败状态只用于调试和测试，不影响 active 播放状态。
```

状态约束：

```text
idle:
  _preloadVideoId == null
  _preloadController == null
  不保留候选、失败目标或可复用资源

initializing:
  _preloadVideoId != null
  _preloadController != null

preloaded:
  _preloadVideoId != null
  _preloadController != null
  _preloadController.value.isInitialized == true

failed:
  _preloadVideoId != null
  _preloadController == null
  只允许由最新 token 的 initialize failure 进入
  表示最近一次目标初始化失败，允许同一 item 下次重试
  不能被当作命中或可复用 preload 资源
```

建议对测试暴露只读 getter：

```dart
String? get preloadVideoId;
bool get hasPreloadController;
bool get isPreloadInitialized;
```

如需更细测试，也可以暴露：

```dart
PreloadControllerStatus get preloadStatus;
```

但该状态仍然是 PlayerController 内部资源状态，不进入 UI 层 `PlayerState`。

## 核心策略

`preloadVideo(item)` 处理流程：

```text
1. 如果 item 是当前 active video，直接跳过。
2. 如果 item 已经是当前 preload video 且状态是 initializing / preloaded，直接跳过。
3. 如果已有旧 preload controller，先释放旧 preload。
4. 递增 _preloadToken，记录新的 _preloadVideoId。
5. 使用默认清晰度创建 preload controller。
6. 调用 initialize。
7. initialize 完成后必须校验 token、controller 引用和 videoId。
8. initialize 成功且仍是最新任务时 setLooping(true)，状态置为 preloaded。
9. initialize 失败且仍是最新任务时释放该 controller，状态置为 failed。
10. 过期任务完成后只能释放自己的 controller，不能覆盖当前 preload 状态。
11. 整个过程不调用 play。
12. 整个过程不修改 PlayerState。
```

本阶段建议默认沿用 active 起播的默认清晰度：

```text
VideoQuality.p720
```

原因：

```text
与当前 playVideo 默认质量一致。
12.4 先验证 preload 生命周期，不引入质量策略变量。
后续如要按网络、设备或用户偏好预加载质量，可以独立扩展。
```

## PlayerController 设计

建议新增字段：

```dart
VideoPlayerController? _preloadController;
String? _preloadVideoId;
VideoQuality? _preloadSelectedQuality;
PreloadControllerStatus _preloadStatus = PreloadControllerStatus.idle;
int _preloadToken = 0;
```

建议保留现有 initialize timeout：

```dart
static const Duration _initializeTimeout = Duration(seconds: 12);
```

`preloadVideo` 伪代码：

```dart
Future<void> preloadVideo(VideoFeedItem item) async {
  if (state.videoId == item.id) {
    return;
  }

  if (_preloadVideoId == item.id &&
      (_preloadStatus == PreloadControllerStatus.initializing ||
       _preloadStatus == PreloadControllerStatus.preloaded)) {
    return;
  }

  await disposePreload();
  final token = ++_preloadToken;

  const selectedQuality = VideoQuality.p720;
  _preloadVideoId = item.id;
  _preloadSelectedQuality = selectedQuality;
  _preloadStatus = PreloadControllerStatus.initializing;

  final source = item.sourceForQuality(selectedQuality);
  final controller = VideoPlayerController.networkUrl(Uri.parse(source.url));
  _preloadController = controller;

  try {
    await controller.initialize().timeout(_initializeTimeout);

    if (token != _preloadToken ||
        _preloadController != controller ||
        _preloadVideoId != item.id) {
      await controller.dispose();
      return;
    }

    await controller.setLooping(true);
    _preloadStatus = PreloadControllerStatus.preloaded;
  } catch (_) {
    if (token == _preloadToken && _preloadController == controller) {
      _preloadController = null;
      _preloadSelectedQuality = null;
      _preloadStatus = PreloadControllerStatus.failed;
    }
    await controller.dispose();
  }
}
```

注意：

```text
preload controller 不添加 _syncFromController listener。
preload controller 不写入 _controller。
preload controller 不写入 _controllerVideoId。
preload controller 不写入 _controllerStartupSession。
preload controller 不更新 PlayerState.isInitializing。
```

`disposePreload` 伪代码：

```dart
Future<void> disposePreload() async {
  ++_preloadToken;
  final controller = _preloadController;
  _preloadController = null;
  _preloadVideoId = null;
  _preloadSelectedQuality = null;
  _preloadStatus = PreloadControllerStatus.idle;

  if (controller != null) {
    await controller.dispose();
  }
}
```

最小过期保护要求：

```text
preloadVideo 每次启动新 initialize 前必须递增 token。
disposePreload 必须递增 token，使未完成的旧 initialize 失效。
initialize success / failure 回来后必须校验 token。
过期 success 不能把 preloadStatus 改为 preloaded。
过期 failure 不能把当前 preloadStatus 改为 failed。
过期任务只能 dispose 自己创建的 controller。
```

这属于 12.4 必须实现的最小异步安全边界。
12.5 再扩展为更完整的快速滑动、资源上限和过期统计策略。

## 与 active controller 的隔离

active 相关字段继续只表达当前播放目标：

```text
_controller
_controllerVideoId
_controllerStartupSession
_lastControllerIsPlaying
_lastControllerIsBuffering
PlayerState
```

preload 相关字段只表达后台准备资源：

```text
_preloadController
_preloadVideoId
_preloadSelectedQuality
_preloadStatus
```

隔离要求：

```text
preload 成功不修改 PlayerState.videoId。
preload 成功不修改 PlayerState.isInitialized。
preload 成功不修改 PlayerState.isPlaying。
preload 成功不修改 PlayerState.wantsToPlay。
preload 成功不触发 VideoPlayerView 绑定。
preload 失败不写 PlayerState.error。
preload 失败不关闭当前 startup session。
preload 失败不暂停或释放 active controller。
```

## 与 playVideo 的关系

12.4 暂不复用 preload controller。

当 `playVideo(item)` 被调用时：

```text
如果 item 命中 preloadVideoId 且 preloadStatus == preloaded：
  12.4 仍然走原 active 初始化流程。
  不 promote preload controller。
  不产生起播性能收益。

如果 item 不命中 preload：
  行为与现状一致。
```

为了避免同一个视频同时被 active 初始化和 preload 持有，建议在 `playVideo(item)` 开头增加防护：

```text
如果即将播放的视频等于 _preloadVideoId：
  先 disposePreload()
  再走原 playVideo 流程
```

这样 12.4 保持语义简单：

```text
preload 只负责提前初始化。
active 播放仍然只来自 playVideo 原链路。
命中 preload 不加速 active 起播。
```

12.6 再把这段逻辑替换为：

```text
命中 preload 时 promote preloaded controller -> active controller。
```

## 与 stop / dispose 的关系

`stop()`：

```text
只停止 active controller。
默认不释放 preload controller。
```

原因：

```text
Feed 当前项可能是图片，但下一条仍是视频。
12.3 的调度语义允许非视频 current item 后继续保留下一条 preload candidate。
```

`stopIfCurrent(videoId)`：

```text
只在 active 命中 videoId 时停止 active controller。
不因为 active stop 自动释放 preload controller。
```

`ref.onDispose`：

```text
必须同时释放 active controller 和 preload controller。
```

建议：

```dart
ref.onDispose(() {
  _closeCurrentStartupSession();
  unawaited(_disposeCurrent());
  unawaited(disposePreload());
});
```

## 与 FeedPlaybackCoordinator 的关系

12.4 不需要改变候选选择逻辑。

现有链路继续成立：

```text
FeedPlaybackCoordinator.handleFeedCurrentChanged(index)
  -> _nextVideoCandidate(items, index)
  -> _schedulePreload(candidate)
  -> PlayerController.preloadVideo(candidate)
```

调度约束：

```text
当前 active 播放流程优先。
preload 调度随后执行。
preload 失败不需要 FeedPlaybackCoordinator fallback。
```

12.4 的 `preloadVideo` 会变成真实异步 initialize。为避免 `handleFeedCurrentChanged` 被后台 preload 长时间阻塞，建议 `_schedulePreload` 后续可改为 fire-and-forget：

```dart
unawaited(playerController.preloadVideo(candidate));
```

但如果测试需要稳定等待 preload 完成，可以保留 `await`，或在测试中直接调用 `PlayerController.preloadVideo` 验证。

本阶段优先保持现有 coordinator 行为不大改，除非测试显示 `await preloadVideo` 明显影响 active 起播链路。

## 失败处理

preload initialize 失败时：

```text
释放失败 controller。
保留 _preloadVideoId 表示最近失败目标。
不写 PlayerState.error。
不影响 active 播放。
不改变 wantsToPlay。
不向 UI 抛异常。
```

推荐失败后：

```text
_preloadController = null
_preloadStatus = failed
_preloadVideoId = item.id
```

原因：

```text
测试和调试可以知道最近失败目标是谁。
下一次 preloadVideo(同一 item) 可以允许重试，或者根据状态 failed 重新初始化。
failed 不是可复用资源状态。
failed 不代表当前仍持有 controller。
```

因此重复请求判断必须只跳过：

```text
initializing
preloaded
```

不能跳过：

```text
failed
```

## 资源释放策略

12.4 只允许：

```text
active controller 最多 1 个。
preload controller 最多 1 个。
```

当新的 preload candidate 到来：

```text
释放旧 preload controller。
再创建新 preload controller。
```

当 candidate 为空：

```text
disposePreload()。
```

当 active 播放目标等于 preload 目标：

```text
12.4 先 disposePreload()，再按原 playVideo 初始化 active。
```

这不是最终性能最优策略，但能保证 12.4 不提前引入 promote 复杂度。

因此 12.4 的效果评估不能写成：

```text
preload 命中后起播更快。
```

只能写成：

```text
preload controller 可以提前初始化并被安全持有。
preload 资源可以被安全释放。
preload 成败不会污染 active 播放状态。
```

```text
preload 完成不会触发 UI 切换。
preload 完成不会修改 PlayerState。
preload initialize 失败不改变当前播放状态。
preload initialize 失败不写 PlayerState.error。

PlayerController dispose 时会释放 active 和 preload 两类资源。

flutter analyze 无新增问题。

```

## 阶段结论

12.4 的重点是把 preload 从“候选 id”推进到“真实已初始化资源”，但仍然保持播放语义克制：

```text
preload 只准备资源。
preload 不代表当前播放项。
preload 不代表用户播放意图。
preload 不绑定 UI。
preload 不自动播放。
```

本阶段完成后，系统应具备：

```text
FeedPlaybackCoordinator 继续决定 current + 1 候选。
PlayerController 可以独立初始化 preload controller。
active 播放状态不受 preload 成败影响。
disposePreload 可以明确释放后台资源。
```

在这个基础上，12.6 才能安全实现真正的 preload hit 热切换。
