# 今日头条视频播放流 + 搜索模块 MVP 框架设计说明文档

## 1. 技术栈

| 类别 | 选型 |
| --- | --- |
| 开发框架 | Flutter |
| 状态管理 | Riverpod |
| 播放能力 | Flutter 视频播放器插件 / 系统播放器封装 |
| 本地存储 | SharedPreferences |
| 数据来源 | 本地 Mock 数据 / 本地缓存 / 搜索离线索引 / 向量检索服务 |


## 2. 功能范围

### 2.1 视频流功能

- App 启动后直接进入内容流页面
- 全屏沉浸式内容流
- 上下滑切换内容
- 视频自动播放
- 视频暂停 / 继续播放
- 播放进度展示
- 拖动进度条调整播放进度

- 视频预加载
- 预加载命中后复用播放器实例
- 清晰度切换
- 横屏播放
- 竖屏恢复
- 视频流中支持图片卡

### 2.2 互动展示功能

- 展示头像
- 展示作者名称
- 展示标题
- 展示点赞按钮

- 视频流顶部展示搜索框
- 点击搜索框进入搜索中间页

- 视频流底部展示相关搜索推荐词
- 推荐词根据当前内容匹配

- 使用推荐词作为搜索关键词发起搜索


## 3. 总体架构设计

项目采用 Flutter + Riverpod 的分层架构。

```text
Page / Widget 页面层
        ↓
Coordinator / Orchestrator 业务编排层
        ↓
ViewModel / Notifier 状态管理层
        ↓
UseCase / Service 业务服务层
        ↓
Repository 数据仓库层
        ↓
DataSource 数据源层
```

### 3.1 分层职责

#### 页面层

负责 UI 展示、用户交互、页面跳转。

#### 业务编排层

负责跨模块业务流程编排，例如内容流当前项变化后的播放 / 停止决策、搜索结果点击后的 Feed 定位与播放器触发、Feed 被搜索页覆盖前后的暂停 / 恢复。

#### 状态管理层

负责页面状态维护、事件处理、异步数据加载。

#### 业务服务层

负责播放器控制、推荐词匹配、搜索匹配、性能统计等业务逻辑。

#### 数据仓库层

负责统一封装数据访问，对上层屏蔽数据来源。

#### 数据源层

负责提供本地 Mock 数据、本地缓存数据、搜索离线业务文档和向量检索结果。

#### 核心服务

```text
RecommendationService
```

#### 推荐词匹配来源

- 当前内容标题
- 当前内容描述
- 当前内容标签
- 当前内容绑定的推荐词

#### 推荐词匹配优先级

1. 优先使用 FeedItem 直接绑定的 recommendationWords
2. 如果没有绑定，则根据 tags 匹配推荐词库
3. 如果 tags 没有命中，则根据 title 关键词匹配
4. 如果 title 没有命中，则返回默认推荐词

### 5.9 performance 模块

- 消费 observability 模块输出的观测数据或 report

#### 核心服务

```text
PerformanceService
```

#### 核心指标

```text
coldStartFirstFrameTime
switchVideoFirstFrameTime
preloadHitFirstFrameTime
preloadMissFirstFrameTime
averageFirstFrameTimeBeforeOptimization
averageFirstFrameTimeAfterOptimization
optimizationRate
```

#### 优化幅度计算方式

```text
optimizationRate =
  (beforeOptimizationTime - afterOptimizationTime)
  / beforeOptimizationTime
  * 100%
```

#### 示例统计结果

```text
未启用预加载：
平均切换起播耗时：850ms

启用预加载并命中 promote 后：
平均切换起播耗时：320ms

优化幅度：
(850 - 320) / 850 * 100% = 62.35%
```

### 5.10 observability 模块

#### 职责

- 旁路记录业务链路中的可观测事件
- 维护起播、预加载、首帧、缓冲等 session 归属
- 处理异步回调迟到、重复事件、失败事件的归属校验
- 输出稳定的调试报告和 baseline report
- 为 performance 模块提供可统计的原始观测数据
- 不参与播放调度决策
- 不直接控制播放器生命周期
- 采集失败不能影响正常业务流程

#### 核心服务

```text
PlaybackStartupMetrics
PlaybackStartupSession
PlaybackStartupReport
```

#### 核心观测事件

```text
feed_item_visible
controller_initialize_start
controller_initialize_end
controller_initialize_failed
first_frame_rendered
play_requested
actual_playing
buffering_start
buffering_end
preload_start
preload_ready
preload_failed
preload_hit
preload_miss
preload_promoted_to_active
preload_disposed_before_hit
```

#### 设计原则

```text
Observability 只能观察业务流程。
Observability 不能决定 active / preload / dispose。
Observability 不能修改用户播放意图。
Observability 不能触发 UI 高频 rebuild。
所有观测事件必须绑定 session / token 并在内部完成归属校验。
迟到事件、重复事件、失败事件必须被安全记录或忽略，不能抛出影响播放链路。
```

## 10. 核心业务流程

### 10.1 App 启动流程

```text
启动 App
  ↓
初始化 Flutter
  ↓
初始化 ProviderScope
  ↓
初始化全局路由
  ↓
进入 FeedPage
  ↓
加载第一页内容流数据
  ↓
初始化首个视频播放器
  ↓
开始播放
  ↓
预加载下一个视频
```

### 10.2 内容流分页流程

```text
进入 FeedPage
  ↓
请求第一页 Feed 数据
  ↓
渲染 PageView
  ↓
用户上下滑动
  ↓
更新 currentIndex
  ↓
判断是否接近列表底部
  ↓
如果接近底部，加载下一页
  ↓
追加到 FeedState.items
```

### 10.3 视频切换播放流程

```text
用户滑动到新内容
  ↓
FeedPage 通知 FeedPlaybackCoordinator 当前项变化
  ↓
判断内容类型
  ↓
如果是图片卡，停止播放器
  ↓
如果是视频卡，触发 PlayerController 播放当前视频
  ↓
检查当前视频是否已预加载
  ↓
命中预加载则直接播放
  ↓
未命中预加载则初始化播放器
  ↓
记录首帧耗时
  ↓
播放当前视频
  ↓
预加载下一个视频
  ↓
取消或释放过期 preload
```

### 10.4 视频预加载流程

```text
当前视频开始播放
  ↓
FeedPlaybackCoordinator 根据滑动方向选择单个候选视频
  ↓
优先选择前进方向或后退方向上的下一个视频项
  ↓
延迟 120ms 调度 preload，快速滑动时取消旧任务
  ↓
PlayerController 创建独立 preload controller 并 initialize
  ↓
标记为 preloaded
  ↓
用户滑动到该视频
  ↓
如果 preload 仍然有效，则 promote 为 active controller
  ↓
直接复用已初始化的播放器实例
  ↓
降低起播耗时
```

### 10.5 清晰度切换流程

```text
用户点击清晰度按钮
  ↓
展示清晰度列表
  ↓
用户选择目标清晰度
  ↓
记录当前播放进度
  ↓
暂停当前播放器
  ↓
```

## 12. 搜索结果匹配策略

### 12.1 搜索范围

```text
SearchVideoDocument.title
SearchVideoDocument.summary
SearchVideoDocument.keywords
VideoFeedItem.id
```

### 12.2 匹配规则

当前默认数据源为 DenseSearchDataSource。

```text
用户搜索词
  ↓
OpenAI-compatible embedding service 生成查询向量
  ↓
MilvusSearchVectorStore 召回向量结果
  ↓
JsonSearchBusinessStore 读取离线业务文档
  ↓
关键词分数与向量分数融合排序
  ↓
根据 videoId 映射回 VideoFeedItem
```

融合分数：

```text
finalScore = vectorScore * 0.7 + normalizedKeywordScore * 0.3
```

关键词分数：

| 命中位置 | 分数 |
| --- | --- |
| title 包含搜索词 | +5 |
| keywords 包含搜索词 | +4 |
| summary 包含搜索词 | +1 |

### 12.3 搜索结果过滤

- 只返回视频类型内容
- 图片卡不出现在搜索结果中
- 融合分数大于 0 的视频进入结果列表
- 结果按照分数从高到低排序
- 如果向量检索、配置或离线业务文档不可用，则 fallback 到 MockSearchDataSource


## 13. 性能优化设计

### 13.1 起播速度优化目标

- 降低首个视频首帧耗时
- 降低上下滑切换视频后的起播耗时
- 降低视频初始化等待时间
- 提升预加载命中率
- 通过数据展示优化幅度

### 13.2 优化手段

- 当前视频优先初始化
- 下一个视频提前预加载
- 复用已预加载的播放器实例
- 快速滑动时取消过期预加载任务
- 图片卡不创建播放器
- 分页数据提前加载
- 封面在播放器初始化前作为占位展示

### 13.3 播放器缓存策略

```text
当前索引：currentIndex
active controller：当前正在播放的视频
preload controller：单槽位，仅保留一个候选视频
候选选择：根据滑动方向优先选择后一个或前一个视频
释放策略：Feed 数据替换、候选失效、开关关闭或新任务覆盖时释放 preload
```

### 13.4 职责边界

```text
FeedPlaybackCoordinator
- 在 currentIndex 变化后编排当前视频播放、方向感知预加载、过期 preload 取消和页面覆盖暂停 / 恢复

PlayerController
- 管理 active controller 和单槽位 preload controller
- 命中 preload 时 promote 为 active controller
- 幂等释放 active / preload 播放器实例

Observability
- 旁路记录 Feed 起播、播放器初始化、首帧、播放确认、缓冲、预加载命中 / 失败等事件
- 维护 session / token 归属，忽略迟到或重复事件
- 不参与 active / preload / dispose 决策
- 不直接控制播放器生命周期

PerformanceService
- 消费 Observability 输出的观测数据或 report
- 统计优化前后起播耗时和优化幅度
- 不直接控制播放器生命周期
```

### 13.5 性能统计指标

```text
videoInitStartTime
videoReadyTime
firstFrameRenderTime
switchStartTime
playStartTime
preloadStartTime
preloadReadyTime
preloadHit
preloadMiss
preloadPromotedToActive
```

### 13.6 优化数据展示

技术文档中展示以下数据：

- 未优化平均起播耗时
- 优化后平均起播耗时
- 预加载命中率
- 预加载提升为 active 的次数
- 平均节省耗时
- 优化幅度百分比

示例：

```text
未优化平均切换起播耗时：850ms
优化后平均切换起播耗时：320ms
平均节省耗时：530ms
优化幅度：62.35%
```

## 17. 架构总结

本项目使用 Flutter + Riverpod 实现，整体采用分层架构。

核心模块包括：

- `feed`：负责内容流展示、分页、视频卡、图片卡
- `player`：负责播放、暂停、进度、清晰度、横屏、预加载和 preload promote
- `search`：负责搜索中间页、搜索历史、搜索结果和混合检索入口
- `feed/coordinators`：负责跨模块业务流程编排，例如 Feed 播放切换、搜索结果定位播放、页面覆盖时的暂停 / 恢复
- `recommendation`：负责相关搜索推荐词匹配
- `observability`：负责播放链路可观测事件采集、session 归属校验和 baseline report
- `performance`：负责起播性能统计和优化幅度计算
- `storage`：负责本地搜索历史和轻量配置存储
- `data`：负责数据模型、仓库和数据源
- `core`：负责全局基础能力
- `common`：负责通用组件

整体数据流保持单向依赖：

```text
UI Page
  ↓
Coordinator / Orchestrator
  ↓
Riverpod ViewModel / Notifier
  ↓
Service / Repository
  ↓
DataSource
```
